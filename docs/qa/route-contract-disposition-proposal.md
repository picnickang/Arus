# Route-Contract Backlog — Disposition Proposal

> **DRAFT — pending owner sign-off.** This document proposes a decision per
> user surface for the **128** remaining route-contract baseline entries
> (`scripts/route-contract-baseline.json`). It changes nothing by itself:
> no UI is removed and no route is added until a row is signed off.
> Per-entry evidence (probe results, file locations) lives in
> [`route-contract-triage.md`](./route-contract-triage.md); counts below are
> by first path segment of the current baseline.

## Recommendation legend

- **Implement-cheap** — a backing implementation already exists (the entry is
  a path/param mismatch, or a thin handler over existing repositories).
  Hours to low-days per surface.
- **Retire UI** — no backend exists and the surface duplicates a working
  feature or shipped design-ahead-of-backend. Deleting the consuming UI
  resolves the entries and ratchets the baseline down.
- **Design-required** — blocked on a product or architecture decision;
  writing code first would bake in the wrong answer.

## Suggested decision order (cheapest, highest-leverage first)

1. **RMS repoint** — five entries, pure client fix, the shore-monitoring page starts working.
2. **Sensor-configuration CRUD** — two entries, small handlers, and it unlocks the ingest-config feature end to end.
3. **STCW + Logbook path reconciles** — compliance-facing pages, backends exist.
4. **Org/user management product call** — biggest UX hole; one decision unblocks six entries (or reopens ADR-002 knowingly).
5. **Analytics retire sweep** — the single largest baseline reduction (up to 17 entries) for deletion-only work.
6. Everything else per the table.

## What sign-off looks like

Mark a row **approved-implement**, **approved-retire**, or **needs-discussion**
(edit this file or reply on the PR). Implementation/removal then proceeds
surface-by-surface, each shrinking the ratcheted baseline — same workflow as
triage passes 1–4 in `route-contract-triage.md`.
