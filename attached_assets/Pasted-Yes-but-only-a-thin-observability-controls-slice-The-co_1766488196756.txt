Yes — but only a thin “observability + controls” slice. The core PdM UI doesn’t need to change.

Here’s exactly what should be reflected in the front end (and what doesn’t need to).

What the front end should add

1) Agent Health panel (new)

You already listed this, and it’s the main UI change:
	•	Connection state: gRPC / Heartbeat / Offline
	•	Frames written, last frame age, agent queue depth
	•	Disk usage (SQLite file size)
	•	Active sources list + Start/Stop controls (even if stubbed for now)
	•	A clear banner if:
	•	agent is down
	•	no frames arriving (stale lastFrame)

This is the “shipboard operator” view.

2) Bridge status (new, minimal)

Because your pipeline is now “agent → SQLite → bridge → Postgres”, you want a quick read on whether the bridge is keeping up:

Expose from Node (easy):
	•	bridge.lastSuccessAt
	•	bridge.cursorLastId
	•	bridge.lagFrames (raw_frames max(id) - cursor.last_id)
	•	bridge.retryBackoffMs / “PG offline” state

Then show in the same panel:
	•	“Bridge: OK / Catching up / PG offline”
	•	“Backlog: 12,450 frames”

This prevents “telemetry is recording but not syncing” confusion.

3) Data freshness indicator on existing dashboards

On any existing “live telemetry” views, add a subtle:
	•	“Last update: X seconds ago”
	•	If stale, show “Data stale” badge

This helps when the vessel is offline or sources are stopped.

What does NOT need to change in the front end
	•	Your existing dashboards/charts logic doesn’t need to know about SQLite or raw frames.
	•	No need to stream raw telemetry into Electron.
	•	No need to rewrite views around the new architecture.

Minimal API endpoints to support the UI

Add these on your Node backend:
	1.	GET /api/agent/status
Returns gRPC-first, heartbeat fallback.
	2.	GET /api/agent/sources
Uses gRPC ListSources.
	3.	POST /api/agent/sources/:id/start and /stop
Calls gRPC Start/Stop.
	4.	GET /api/bridge/status
Reads cursor + computes lag + reports last success + current backoff.

That’s it.
