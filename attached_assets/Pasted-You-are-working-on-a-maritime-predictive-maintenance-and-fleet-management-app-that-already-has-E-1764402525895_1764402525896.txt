### 1. AQM/FMCC INTEGRATION DESIGN (BACKEND)

Design a small, modular **Aquametro FMCC integration layer** that can work in both cloud (Postgres) and vessel/embedded (SQLite) modes.

Implement in a clear folder, for example:

- `server/integrations/aquametro-fmcc.ts` (or `.js`)
- plus any config/types under `server/integrations/` or `server/config/`

The integration must:

1. Support both:
- **REST API** (if vessel has the Aquametro network module)
- **Modbus TCP** (optional, if used – if there is no Modbus client in the project, stub the interface but don’t break anything)
2. Use **config-driven setup**, so we can enable/disable per environment/vessel:
- Global env keys:
- `FMCC_ENABLE` (boolean)
- `FMCC_PROTOCOL` = `"rest"` | `"modbus"`
- If REST:
- `FMCC_BASE_URL`
- `FMCC_API_KEY` or credentials if needed
- If Modbus:
- `FMCC_MODBUS_HOST`
- `FMCC_MODBUS_PORT`
- Optional `FMCC_MODBUS_UNIT_ID`
3. Provide a minimal **service API** to the rest of the app, with well-typed methods such as:

- `getInstantFuelFlow({ vesselId, at }): Promise<{ foFlowKgPerH, doFlowKgPerH, density, temperature, returnFlow, meterStatus }>`
- `getCumulativeFuelCounters({ vesselId, from, to }): Promise<{ foConsumedKg, doConsumedKg }>`
- `getEngineFuelEfficiency({ vesselId, from, to }): Promise<{ sfoC_gPerKWh, loadProfile, rpmProfile }>`
- `getRawStreamSample(...)` for debugging (optional).

Implement appropriate **error handling** and timeouts. If FMCC is disabled or unreachable, the service should fail gracefully and return a clear "integration disabled/offline" error instead of crashing the app.
