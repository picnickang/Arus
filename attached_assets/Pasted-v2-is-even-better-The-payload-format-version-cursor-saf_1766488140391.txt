Here are the last “ship it” refinements I’d apply (still small, but they prevent the most common real-world failures):

1) Make writeBatch() fast enough

A per-reading await client.query(...) loop will be too slow at scale. Keep the same semantics, but implement bulk insert.

Use one of these patterns:

Option A (recommended): COPY into a temp table + upsert
	•	fastest for thousands of rows
	•	still can do idempotency via unique constraint

Option B: single INSERT with many VALUES

Good enough up to a few thousand rows per batch.

Pseudo:

INSERT INTO equipment_telemetry (col1, col2, ...)
VALUES
  ($1,$2,...),
  ($n,$n+1,...),
...
ON CONFLICT (idempotency_key) DO NOTHING;

This change alone will 10× throughput.

2) Define what “writeBatch” means for metrics + retries

Since TelemetryBatchWriter already has Prometheus metrics and retry behavior, don’t lose that visibility.

Add:
	•	arus_bridge_batch_size
	•	arus_bridge_batch_commit_ms
	•	arus_bridge_retry_count
	•	arus_bridge_last_success_unix_ms

If you already have a metrics helper, reuse it.

3) Cursor should advance to the last frame id, not last ts

You’re already doing maxId and maxTs. Keep both for debugging, but correctness is by id.

Also store:
	•	updated_at = nowMs

4) Ensure payload_format_version is carried through

Your Node RawFrame type should include:
	•	payloadFormatVersion: number
and decodeJ1939() should reject unknown versions:

if (frame.payloadFormatVersion !== 1) return [];

That prevents silent mis-decodes later.

5) Agent cursor-safe retention needs access to cursor

Your agent retention references GetCursorLastId(). Since cursor is in SQLite and owned by the bridge, you have two clean choices:
	•	(Simplest): Agent reads ingest_cursor table too (read-only) and uses that last_id.
	•	Or agent exposes gRPC GetBridgeCursor() (but that’s coupling).

I recommend: agent reads the cursor table (read-only). It keeps boundaries clean and removes IPC dependency.

6) Bridge should not hammer Postgres during outages

Add exponential backoff capped at, say, 30 seconds:
	•	2s → 4s → 8s → 16s → 30s
Reset on success.

7) “writeBatch” and “flushOnce” can share code

To avoid duplication inside TelemetryBatchWriter:
	•	Implement a private commitReadings(readings) used by both
	•	Keep queue() path unchanged, but when it flushes it calls the same commit function.
