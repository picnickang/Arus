**RAG Access Plan**
This is the one I would not approve as written.

Issues:
- Title says “Optional” and line 4 frames it as a within-company privacy gap. That is now known false: it is a tenant data-leak path.
- Line 10 makes adding `org_id` to `rag_messages` out of scope. That is acceptable for a hotfix, but the plan should explicitly say this is a phase-1 route/service guard, with `rag_messages.org_id` as a follow-up hardening item.
- It only calls out messages, PATCH, DELETE. The code also has stream history loading via `conversationService.getMessages(conversationId, 10)` and export paths that should be checked.
- It should remove or bypass `DEFAULT_ORG_ID` fallbacks for authenticated RAG conversation routes, not just add owner checks.

Change the plan to:
- Rename to “RAG Tenant Isolation Hotfix”.
- Done criteria: all conversation reads/writes/messages/export/stream-history verify `conversation.orgId === req.orgId` before touching messages.
- Add a service helper like `getConversationForOrg(conversationId, orgId)` and make `getMessages` either require `orgId` or only be called after a checked parent load.
- Prefer `404` for unauthorized conversation IDs unless product explicitly wants `403`.
- Add tests for cross-org denied, same-org allowed, and stream/export denied.

**Migration Runner Plan**
This is a good plan, but step 1 needs sharper wording.

The risk is not just “choose `scripts/run-sql-migrations.mjs`.” That runner applies root `migrations/NNNN_*.sql`, while `server/scripts/migrate.ts` also handles Drizzle plus `server/migrations/*.sql`. A canonical deploy runner must apply every required family or explicitly retire the others.

Tighten done criteria:
- One command covers root SQL migrations, any Drizzle-tracked migrations still required, and `server/migrations/*.sql`, or documents why one family is obsolete.
- `MIGRATE_ON_BOOT`, `db:migrate`, `db:migrate:deploy`, and post-merge use the same runner path.
- Status output reports all migration ledgers, not just one.
- Add object assertions for `0024` telemetry dedup, hot-path indexes, and FK cascade rules.

Also remove “staying single-tenant by design” language from line 12. It is fine to keep RLS changes out of scope, but don’t encode the wrong product assumption.

**Telemetry Bulk Insert Plan**
This is a good, contained performance plan. It correctly depends on the dedup index from the migration task.

Add a few implementation constraints:
- Preserve current retry semantics in `TelemetryBatchWriter`: one chunk failure must requeue/drop the same way as today.
- Use `onConflictDoNothing` or equivalent against the natural key from `0024`, and do not accidentally make duplicates fatal.
- Decide whether callers need inserted row count vs attempted row count. If counters currently imply “accepted,” bulk insert should return enough information to keep metrics honest.
- Keep chunk size configurable. A 500-row multi-value insert is probably fine, but telemetry metadata JSON can make statements large.

I would do this after migration runner unification, not before.

**Offline Conflict Detection Plan**
The plan points at the right problem, but it is too broad for safe execution as written.

Missing decisions:
- Which tables are conflict-enabled? Do not implement arbitrary dynamic table update from request `table`. Use an allowlist with table metadata.
- Which fields are safety-critical and require manual resolution?
- What is the exact version contract: client version equals server version, server increments version transactionally, conflict if stale?
- Should routes use `req.orgId` only? Current `/api/sync/check-conflicts` accepts `orgId` in the body, and `/auto-resolve` queries conflicts without an org filter. The plan should explicitly fix both.
- Is `sync_conflicts` covered by tenant RLS/tenant table registry? If not, add it or ensure every query is org-scoped.

Suggested phase 1:
- Implement only `work_orders` or another high-value table end to end.
- Add optimistic `WHERE id = ? AND org_id = ? AND version = ?` update.
- On mismatch, persist conflict and pause replay.
- Then broaden table coverage.

**ML Training Endpoints Plan**
This is a good “make it real or honest” plan. I’d bias toward the honest-response option first because it is quick and removes misleading behavior without committing to a duplicated training architecture.

Tighten it:
- Current routes accept `body.orgId ?? req.orgId`; training should not let request body override authenticated org.
- Training endpoints should be admin/maintenance gated, not just generally rate-limited.
- If choosing “honest,” return `501 Not Implemented` or `503 Training unavailable`, with a stable machine-readable code, not a 200.
- If choosing “real,” reuse the model registry/training pipeline path and expose `runId`, status polling, artifact digest, dataset hash, params, metrics, and failure reason.

**Recommended Next Step**
Revise the RAG plan first and execute it as a small security patch. The other four are actionable once their acceptance criteria are tightened, but they should not block the RAG fix.
