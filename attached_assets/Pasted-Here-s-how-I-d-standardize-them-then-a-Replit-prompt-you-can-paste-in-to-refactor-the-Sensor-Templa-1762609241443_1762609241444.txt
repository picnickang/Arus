2. Simplify the “Create Sensor Template” dialog

In the create-template form:

Replace free-text kind with a <Select> whose options are the keys of SENSOR_KIND_PRESETS.

When the user selects a kind:

If unit is empty, set it to SENSOR_KIND_PRESETS[kind].defaultUnit.

If fields JSON is empty or {}, set it to the defaultFields object.

Keep the “Fields (JSON)” textarea, but:

Prepopulate it with JSON.stringify(defaultFields, null, 2) on first selection.

Validate that it’s valid JSON on submit; show a friendly error if parsing fails.



3. Auto-generate templateId (optional but recommended)

In the template form:

Make name required and templateId optional.

On submit, if templateId is blank:

Generate one like:

KIND-NAME-SHORT-SLUG-###

where KIND is uppercased sensor kind (e.g. OIL_DEBRIS) and ### is a short random suffix.

Ensure this is done on the client before the POST to /api/sensor-templates.



4. Simple vs Advanced mode

Add a small toggle above the JSON field:

const [advancedMode, setAdvancedMode] = useState(false);
Simple mode (default):

Show kind, name, unit.

Hide raw JSON textarea.

Render read-only pretty view of the default fields as a small table:

Parameter | Value | Unit (if applicable)

Advanced mode:

Show the raw JSON textarea (editable).

Allow direct editing of fields.



Use a Switch or a checkbox labeled “Advanced (edit raw JSON)”.



5. Make empty state friendlier

In the Sensor Templates list:

If there are zero templates:

Show text:

“No sensor templates yet. Create an oil debris, vibration, or temperature template to get started.”

Keep the “+ Create Template” button.

Optionally: add a second button “Create Oil Debris Starter” that opens the create dialog pre-selected to kind = "oil_debris".



6. Type safety

Wherever you refer to kind in template types, narrow it:

If there is a SensorTemplate interface/type, set kind: SensorKind (instead of plain string) if possible without breaking the API layer.



Constraints

Do NOT change the backend schema or routes in this task; assume /api/sensor-templates already persists kind, unit, and fields as today.

Do NOT break existing code that consumes sensor templates; keep the fields JSON shape backward compatible (it just gets richer defaults).

Make sure the UI still works if an unknown kind comes back from the server: fall back to generic behavior (no defaults, just show it as plain text).



Acceptance checklist

The “Kind” dropdown only shows: vibration, pressure, temperature, flow, level, voltage, current, frequency, rpm, oil_debris, acoustic, position.

Selecting a kind auto-fills unit and fields for new templates.

Simple mode hides raw JSON and shows a friendly preview of fields.

Advanced mode exposes a JSON textarea, with validation on submit.

Creating a new oil_debris template feels smooth: choose kind → name → save, with no need to manually craft JSON.



Apply all of these changes now.
