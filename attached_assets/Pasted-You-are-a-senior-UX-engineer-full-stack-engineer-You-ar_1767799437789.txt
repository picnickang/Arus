You are a senior UX engineer + full-stack engineer. You are working inside an existing Node.js/TypeScript app (ARUS). The PdM “Schedule” tab already exists (mobile view shows KPI tiles + vessel x day grid). Your task is to keep the existing visual style, but implement correct Schedule UI/UX behavior so it works as an operational planning tool.

Do NOT redesign the whole app. Improve function and interactions while staying consistent with current components.

====================================================
1) CORE UX GOAL (WHAT THIS PAGE IS FOR)
====================================================
This Schedule tab is the “action layer” of PdM:
- It turns PdM alerts (risk + RUL + evidence) into a proposed maintenance plan.
- Users must be able to: see what’s planned, see what’s blocked, understand why, and create/track work orders.

====================================================
2) REQUIRED UX BEHAVIORS (MUST IMPLEMENT)
====================================================

A) Global filters drive everything
- Fleet/Vessel filter
- Equipment type filter
- Date range/week selector
Behavior:
- Any change triggers refetch of schedule data and re-renders KPIs + grid + unassigned list.
- Persist filter state in URL query params (preferred) and/or local storage.

B) Two tabs must be coherent
- Risk Queue = “what is risky”
- Schedule = “what we plan to do”
Behavior:
- From Risk Queue, user can “Add to Schedule” (or “Propose Slot”) which creates a schedulable task.
- From Schedule, user can tap a scheduled item to view the original PdM alert evidence.

C) KPI tiles are actionable
- “Scheduled This Week” tile should be tappable:
  - opens a filtered list of scheduled tasks for this week.
- “Unassigned High-Risk” tile should be tappable:
  - opens the blocked/unassigned list, sorted by urgency and latest safe finish.

D) Grid is a planning tool, not a static table
- Rows = vessels (or optionally vessel groups)
- Columns = days in selected week
- Each cell contains scheduled task “chips/cards”
Each chip shows (compact):
- severity color
- asset short name
- failure mode short label
- duration (optional small badge)
- status icon (draft/approved/in-progress)

E) Selection + detail drawer (critical UX)
When user taps a task chip:
- Open a bottom sheet (mobile) / side panel (desktop) with Task Detail.
Task detail MUST include:
- Asset + vessel header
- Failure mode label
- RUL range (P10–P90) and confidence badge
- Evidence chips (why)
- Recommended actions checklist
- Scheduling window:
  - earliest start
  - preferred date
  - latest safe finish (“must finish by”)
- Status + work order linkage (if present)
Actions in detail:
- Create Work Order Draft (if available)
- Mark Scheduled / Unschedule
- Escalate (if blocked/high risk)
- Add note (optional)
- View PdM Alert (deep link)

F) Drag/drop or quick move (Phase 1 minimal)
If drag/drop is feasible with existing UI libs:
- Allow dragging a task chip to another day cell within allowed window.
If drag/drop is NOT feasible:
- Provide “Move” action in detail drawer:
  - shows allowed dates (within earliest → latest)
  - user selects a new date
Rules:
- If moved outside allowed window, show confirmation and mark as “override”.

G) Blocked / Unassigned section is required (cannot hide)
If tasks cannot be scheduled, they must show clearly with reasons.
UI:
- A collapsible panel below grid: “Unassigned / Blocked”
Each item shows:
- severity
- asset + vessel
- latest safe finish date
- block_reason (capacity/parts/availability/telemetry stale/lead time)
Actions:
- “Request override” (just marks override)
- “Increase capacity” (links to config/settings if exists)
- “Create WO anyway” (optional and gated)
- “View alert”

H) Empty states must be informative
If no tasks:
- Show: “No PdM tasks in this week for current filters.”
If tasks exist but none scheduled:
- Show: “Tasks exist but none could be scheduled.” + open Unassigned panel.
If telemetry stale for many assets:
- Show banner: “Telemetry delayed on X vessels; schedule confidence reduced.”

I) CSV export matches what user sees
- Export uses current filters + current week.
- Include scheduled tasks and unassigned tasks.
Columns:
- vessel, asset, failure_mode, severity
- rul_p10/p50/p90
- earliest/preferred/latest
- scheduled_start/end
- status
- block_reason
- work_order_id (if any)

====================================================
3) UX STATES + DATA LOADING (IMPORTANT)
====================================================
A) Loading states:
- When filters change, show skeletons for KPIs and grid.
- Keep previous data visible with a subtle “updating” indicator (optional).

B) Error states:
- If schedule endpoint fails:
  - show toast + inline “Retry”
  - preserve last known data

C) Real-time refresh (optional, but recommended):
- Poll every 60–120 seconds OR refresh on tab focus.
- Avoid disrupting the user if a detail drawer is open.

====================================================

====================================================
- The Schedule tab should deep-link:
  /maintenance/predictive-maintenance?schedule=1&week=YYYY-WW&vesselId=...

  /maintenance/predictive-maintenance?schedule=1&taskId=ALERT_ID

- “Ingestion Health” should navigate to the ingestion diagnostics route.

- grid {dates, rows, cells}
- flat task list (for detail + export)
- unassigned list with reasons
