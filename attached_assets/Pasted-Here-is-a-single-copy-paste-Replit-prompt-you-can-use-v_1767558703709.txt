✅ Single Replit Prompt

Goal: Safely refactor a large monolithic TypeScript database schema (~10k LOC, 222 tables, 468 dependent imports) into an internally modular structure without changing any existing import paths or breaking frontend/backend consumers.

Instructions:

You are refactoring an existing TypeScript schema that currently lives in a single monolithic file and is imported across hundreds of files.
This schema is intentionally shared between frontend and backend and acts as a central contract/type registry.
You must preserve all public exports and existing import paths.

Hard constraints (must follow):
	1.	DO NOT change any existing import paths in the codebase
	2.	DO NOT change exported names or shapes
	3.	DO NOT introduce breaking runtime behavior
	4.	DO NOT apply hexagonal or domain logic to the schema layer
	5.	Assume 400+ dependent imports exist — backward compatibility is mandatory

⸻

Refactor Strategy (Required)

1. Create an internal modular structure
Split the schema internally by bounded context while keeping a single public entrypoint.

Create this folder structure:

src/schema/
  core/
  shared/
  maintenance/
  crew/
  inventory/
  workorders/
  relations/
  index.ts

	•	Move table definitions into appropriate domain folders
	•	Keep “core” and “shared” for cross-cutting tables (users, tenants, enums, reference tables)

⸻

2. Separate table definitions from relations (critical)
To avoid circular dependencies:
	•	Table files MUST define only tables, columns, and enums
	•	No foreign key or cross-table imports inside table files
	•	Move all ORM relations / foreign key wiring into:

src/schema/relations/

Relations files may import multiple tables and define associations centrally.

⸻

3. Preserve the public API (no breaking changes)
src/schema/index.ts must:
	•	Re-export everything that was previously exported by the monolithic schema file
	•	Act as a façade so existing imports continue to work:

export * from "./core";
export * from "./shared";
export * from "./maintenance";
export * from "./crew";
export * from "./inventory";
export * from "./workorders";
export * from "./relations";

No consumer code should need to change.

⸻

4. Reduce circular dependency risk
	•	Ensure domain schema modules do not import each other directly
	•	All cross-domain references must occur only in relations/
	•	Use lazy or deferred relation registration if supported by the ORM

⸻

5. (Optional but recommended) Prepare for dual entrypoints
If feasible without breaking changes:
	•	Keep runtime ORM objects available for backend
	•	Prepare for a future types-only export surface (pure TS types, no runtime ORM dependency)

Do not enforce this yet—only structure the code so it’s possible later.

⸻

Output Expectations
	•	Produce the new folder structure
	•	Move code incrementally (do not rewrite logic)
	•	Keep formatting and naming consistent
	•	Leave TODO comments where domain ownership is ambiguous
	•	Add brief comments explaining why relations are centralized

⸻

Success Criteria
	•	App compiles with zero import changes
	•	Schema is internally modular and readable
	•	Circular dependencies are reduced or eliminated
	•	Frontend and backend continue using the schema as a shared contract
