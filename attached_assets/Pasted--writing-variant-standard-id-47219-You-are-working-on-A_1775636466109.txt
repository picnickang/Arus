You are working on ARUS, a React + TypeScript fleet operations platform.

Your task is to fix the certification ↔ equipment UX/data gaps across Crew, Equipment, Scheduling, and Compliance.

This is not a discovery task.
This is an implementation task.

Be practical, ship sensible defaults, reuse existing APIs where possible, and add TODO comments where backend changes are still needed.

After implementation, perform a full CRUD test of the new certification flows and report the results.

⸻

GOAL

Make certifications a first-class operational concept in the UI.

Users should be able to answer these questions easily:
	•	Does this equipment have valid vessel/equipment certificates?
	•	Which certificates cover this equipment?
	•	Which crew are qualified for a shift on this equipment?
	•	Are any required certifications expiring soon?
	•	Are shift certification requirements standardized and trustworthy?

⸻

CURRENT PROBLEMS TO FIX
	1.	Vessel/equipment certificates exist in the backend API but there is no frontend page
	2.	Equipment screens do not show certification status
	3.	Shift template certRequired is free text and causes inconsistent data
	4.	There is no visible bridge between:
	•	crew certifications
	•	shift requirements
	•	equipment certification requirements
	5.	Crew “Documents” and crew certifications feel like overlapping concepts

⸻

4. Build the scheduling qualification bridge

Add a visible UI that connects:
	•	shift requirement
	•	equipment
	•	crew qualifications

At minimum implement one of these:

Option A: Shift detail panel
When viewing a shift/template, show:
	•	required certification(s)
	•	matching eligible crew
	•	crew with expiring certs
	•	crew missing required certs

Option B: Qualification preview in shift template form
When configuring a shift template:
	•	show required cert
	•	preview how many crew currently match it
	•	warn if requirement is rare / expiring / unmatched

Goal

Managers should be able to ask:
	•	who is qualified for this shift?
	•	is this shift at risk because of certification gaps?

⸻

5. Clarify Crew Documents vs Certifications

Do not leave the overlap confusing.

Preferred solution

In crew profile, rename / restructure the tab or section to:

Documents & Certifications

Then split content into sub-sections:
	•	Travel / Identity Documents
	•	Medical Documents
	•	Professional Certifications

Minimum acceptable solution

If structural unification is too large right now, rename labels so the distinction is obvious:
	•	Documents → Travel & Identity Documents
	•	Certifications → Professional Certifications

Goal

Users should not wonder whether STCW belongs under “Documents” or somewhere else.

UX REQUIREMENTS

Use consistent status language everywhere

Preferred statuses:
	•	Current
	•	Expiring Soon
	•	Expired
	•	Missing

Avoid inconsistent labels across pages.

Show relationship both ways

Users should be able to navigate:
	•	equipment → certificates
	•	certificate → covered equipment
	•	shift → required certification
	•	certification → eligible crew (or at least matched crew count)

No hidden relationship

If the system knows a certification relationship, the user should be able to see it.

Keep forms clean

Do not create giant forms.
Use sections, clear labels, and controlled inputs.
