2. WEAK POINTS & RISK AREAS

Next, identify specific weak points, focusing on:
	1.	Remaining “god files”
	•	List any files still over ~400–500 lines.
	•	For each, briefly state:
	•	Role (e.g., “mega React page for X”, “giant route handler”, “huge service”).
	•	What sub-modules it should be split into (e.g., hooks, subcomponents, service helpers).
	2.	Potential crash / lag risk
	•	In telemetry ingestion & processing:
	•	Any unbounded loops, synchronous heavy operations, or missing back-pressure.
	•	In React:
	•	Very large tables or dashboards rendered without virtualization or pagination.
	•	In database access:
	•	N+1 query patterns, unindexed filters, or blocking calls in main Electron process.
	3.	Data Path Confusion / Isolation
	•	Where recent modularization may have accidentally isolated data flows (e.g., things that used to be wired directly now don’t talk to each other).
	•	Check: do work orders, inventory, logbooks, and telemetry still connect where intended?
	4.	Schema / Migration Risks
	•	Any tables used by code with missing migration or vice versa.
	•	Any field mismatches between backend, frontend, and exports (PDF/Excel).
