You are assessing ARUS for production launch readiness. ARUS is a maritime 
fleet management and predictive maintenance platform being prepared for first 
deployment to ARUS Sdn Bhd, a Brunei offshore vessel operator. It sits on top 
of SHIPMATE (in production since 1995) as an AI/analytics overlay.

Your job is to produce a LAUNCH READINESS ASSESSMENT, not a code review. 
The goal is to identify what actually blocks deployment, separately from what 
is aesthetic debt that can be deferred.

## Ground rules

1. Be blunt. If something looks like it won't survive contact with a real 
   vessel environment, say so. Do not soft-pedal risk.
2. Distinguish clearly between:
   - P0 blockers: deployment fails or user flow is broken
   - P1 risks: deployment works but something will likely break in first 
     30 days under real usage
   - P2 concerns: known debt, fine to ship, track for v1.1
   - Aesthetic: TS errors, unused exports, dead code — explicitly OUT OF SCOPE 
     for this assessment unless they cause runtime failures
3. Favor concrete observations over abstract critique. "X file has Y problem 
   at line Z" beats "code quality is mixed."
4. If you cannot verify something without running it, say so and tell me 
   what to run.

## What to assess

For each of the areas below, answer: (a) is this ready to ship to ARUS Sdn Bhd 
as v1.0? (b) what's the specific concern if not? (c) what would it take to 
make it ready?

### 1. Installer & deployment
- Windows installer: does it actually install on a clean Windows Server 
  2016/2019/2022 and Windows 10/11? Flag any OS-version assumptions.
- Service account creation, permissions, DenyLogon setup
- Database migration on install vs upgrade
- First-run setup wizard flow
- Uninstall/reinstall paths
- Vessel vs cloud installer variants — are both known-working?
- Review the prior installer CRUD test findings. Which are resolved, which 
  are not?

### 2. SHIPMATE integration
- Import adapter: does it handle real SHIPMATE exports, not just test data?
- Schema mismatch handling between SHIPMATE's 1995-era model and ARUS
- Data volume: what happens with 10+ years of historical PMS records?
- Error recovery if an import fails partway through
- Rollback: if ARUS writes to its own tables from a SHIPMATE import, can 
  that be undone?

### 3. Offline-first / sync
- The vessel-to-cloud sync system (syncJournal, syncOutbox) — have extended 
  offline periods (72+ hours) been actually tested?
- Conflict resolution: what happens when a work order is edited on both 
  vessel and cloud during offline?
- Sync queue overflow: what's the bounded behavior if the outbox grows to 
  100K+ entries?
- Does the vessel-side SQLite schema actually work end-to-end, or has 
  testing been primarily on the PG/cloud side?

### 4. Core user flows (smoke-test style)
For each of these, trace the full path from UI click to DB write and back. 
Flag anything that looks incomplete, buggy, or fragile:
- Login / auth / session
- Create work order → assign crew → add parts → complete → log labor
- View vessel dashboard
- Run a predictive maintenance alert workflow
- Import SHIPMATE data
- Generate a scheduled report
- Crew rest hours entry + compliance check
- Receive an alert email (end-to-end: trigger → email_settings → SMTP → delivery)

### 5. Operational readiness
- Are there runbooks for ARUS Sdn Bhd operations staff? If not, who writes them?
- Backup/restore procedure on the vessel side — documented and tested?
- Logging and diagnostics — if something breaks in Brunei, can we triage 
  remotely?
- Auth/licensing setup for the customer — do I have a documented process?
- Known-good initial data seed for a new vessel

### 6. Security basics
- Secrets management (SMTP passwords, API keys) — actually encrypted at rest?
- Auth: password policy, session handling, MFA if relevant
- The bcrypt→bcryptjs migration — are there any residual bcrypt references?
- RBAC: does the customer have admin/operator/viewer separation working?
