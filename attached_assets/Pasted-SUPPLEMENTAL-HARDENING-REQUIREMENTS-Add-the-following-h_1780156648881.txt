SUPPLEMENTAL HARDENING REQUIREMENTS

Add the following hardening requirements to the implementation.

1. Audit logging

Add audit logging for sensitive administrative and safety actions.

Audit these events:
- role created
- role edited
- role activated/deactivated
- role deletion attempted
- role deleted/deactivated
- user role assignment changed
- user vessel/department assignment changed
- dashboard configuration changed
- regular user credentials created
- password reset by admin
- login access enabled/disabled
- user changed own password
- alarm type created
- alarm type edited
- alarm type activated/deactivated
- alarm type deletion attempted
- safety alarm activated
- safety alarm updated
- safety alarm cleared
- safety alarm acknowledged

Audit log should include:
- actor user id
- actor username/name if available
- action type
- target type
- target id
- before/after metadata where safe
- timestamp
- IP/session metadata if already available

Never log:
- plaintext passwords
- password hashes
- sensitive secret values

If an audit log system already exists, use it. If not, add a minimal typed service or table.

2. Real-time or near-real-time safety alarm delivery

Safety alarms should be visible quickly to affected users.

Use existing real-time infrastructure if available:
- WebSocket
- SSE
- existing notification bus
- existing polling/query invalidation pattern

If no real-time system exists, add a safe polling fallback on the existing User page for active safety alarms.

The User page should refresh active safety alarms frequently enough that an emergency banner appears without requiring a full manual reload.

Do not overbuild a new messaging system if the app already has one.

3. Alarm drill/test mode

Support alarm mode:

- real
- drill
- test

Drill/test alarms should be visually distinct from real alarms.

The user-facing banner should clearly show:
- “DRILL”
- or “TEST”
when applicable.

Drill/test alarms should still respect vessel/fleet scoping.

Admin should confirm before activating a real emergency or critical alarm.

4. Multiple-role conflict resolution

If a user has multiple roles, implement clear dashboard merging rules.

Preferred behavior:
- combine enabled widgets across active assigned roles
- remove duplicates
- apply the most restrictive visibility scope unless the user has explicit fleet/department permission
- show role labels where a widget is included because of a specific role
- never grant admin-only dashboard data merely because a widget is enabled unless the user has the required permission
- if role configs conflict, server-side authorization wins

Support a primary role if existing user/crew model allows it.

If a primary role exists:
- use primary role for dashboard title/preview
- still merge allowed widgets from secondary roles where safe

5. Credential and login security

Regular user credential management must be secure.

Requirements:
- passwords must be hashed using the existing secure auth mechanism
- plaintext passwords must never be stored
- plaintext passwords must never be returned after creation/reset
- password hashes must never be returned by API
- password values must never be logged
- add rate limiting to regular user login if the app has rate-limiting middleware
- add lockout or temporary delay after repeated failed login attempts if the auth system supports it
- when admin resets a password, require password change on next login by default
- when password is reset, invalidate existing sessions if the session system supports it
- when user changes password, update passwordChangedAt and clear mustChangePassword

6. Admin lockout protection

Prevent destructive changes that could lock admins out.

Rules:
- do not allow deletion/deactivation of the last admin-capable role
- do not allow removal of admin role from the last active admin user
- do not allow disabling login for the last active admin user
- protected admin role cannot be hard-deleted
- return a clear 409 response when an action would cause admin lockout

7. Safety alarm acknowledgement accountability

If acknowledgement is implemented, track:
- alarm id
- user id
- acknowledgedAt
- acknowledgement source/device if available
- optional comment if supported

Admin Safety tab should show:
- total affected users
- acknowledged count
- not acknowledged count
- list of unacknowledged users if practical

Regular User page should:
- show acknowledge button only when acknowledgement is required
- prevent duplicate acknowledgement
- show acknowledged state after submission

8. Seed data and migration safety

If adding migrations, seed protected defaults.

Protected roles:
- admin
- chief_engineer
- technician
- vessel_master
- crew_member
- logistics_user
- procurement_user
- safety_officer
- maintenance_planner

Protected alarm types:
- fire_alarm
- man_overboard
- abandon_vessel
- medical_emergency
- collision_grounding
- flooding_water_ingress
- engine_room_emergency
- security_threat
- gas_leak
- machinery_emergency
- evacuation
- muster_alarm
- general_emergency

Migrations should be idempotent where practical.
Do not duplicate seed rows if migration or seed runs more than once.

9. Operational safety note

The in-app safety alarm system is an operational communication layer.

It must not be presented as a replacement for:
- physical vessel alarm systems
- muster procedures
- radio communication
- SMS/phone escalation where required
- regulatory emergency procedures
- company safety management system requirements

Use wording such as:
“In-app emergency notice” or “Safety alarm notification” where appropriate.

10. Failure states

Handle failure states clearly.

If active alarm fetch fails:
- do not block the whole User dashboard
- show a warning that emergency status could not be refreshed if appropriate
- log the error server-side

If admin alarm activation fails:
- show clear error
- do not show success toast unless the backend confirms activation

If dashboard config fetch fails:
- fall back to a safe minimal user dashboard
- do not expose admin data
- show safety alarms independently if possible

11. Observability

Add structured logs or existing telemetry events for:
- alarm activated
- alarm cleared
- alarm acknowledgement submitted
- failed alarm activation
- failed permission check on safety/role APIs
- failed login attempt if existing auth logging supports it

12. Acceptance criteria

Before marking complete, verify:

- Admin can create/edit/deactivate roles.
- Admin cannot delete protected roles.
- Admin cannot delete assigned active roles.
- Admin can configure role dashboards.
- Existing User page renders based on role config.
- Multiple-role users do not receive unauthorized data.
- Admin can create/edit/deactivate alarm types.
- Admin must select alarm type before activation.
- Admin can activate Fire Alarm for Vessel Alpha.
- Vessel Alpha user sees alarm at top of User page.
- Vessel Beta-only user does not see Vessel Alpha alarm.
- Fleet-wide alarm appears for relevant users.
- Alarm can be cleared and disappears from regular User page.
- Passwords are hashed and never displayed after save.
- User can change password and admin sees updated metadata.
- Audit log records sensitive actions.
- `npx tsc --noEmit` passes.
