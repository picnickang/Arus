You are a senior product/UX/operations analyst evaluating the financial system design for ARUS, a fleet operations and predictive maintenance platform.

Your job is to evaluate whether the current or proposed financial layer is:
- credible
- useful for operators and managers
- sufficient to prove ROI
- integrated correctly with maintenance workflows
- not turning into a bloated accounting/ERP system

Be direct, critical, and practical.

---

# CONTEXT

ARUS is not meant to be a full accounting platform.

It is a fleet operations / predictive maintenance system with:
- Equipment-centric workflows
- Work Orders (engineer-owned)
- Service Orders / Procurement records (procurement-owned, but linked to Work Orders)
- AI predictions, RUL, telemetry, diagnostics, maintenance scheduling
- Decision support that should recommend actions such as:
  - repair now
  - repair at port
  - request parts
  - create work order

The financial layer is intended to support:
- decision-making
- cost comparison
- ROI proof
- avoided-failure calculations
- maintenance optimization

NOT:
- general ledger
- invoicing
- payroll
- tax/accounting compliance
- full ERP replacement

---

# CORE QUESTION

Evaluate whether ARUS should include a financial layer, and whether the proposed financial model is the right kind of finance:

- embedded operational finance
- not standalone accounting
- connected to Work Orders, Procurement, Equipment, and Decisions

---

# EVALUATION AREAS

## 1. Does the financial system belong in ARUS?
Evaluate whether this product should include:
- repair cost estimation
- failure cost estimation
- downtime cost
- avoided failure cost
- ROI tracking
- estimated vs actual cost

Explain what absolutely belongs here and what should stay outside the system.

---

## 2. Is the object model correct?
Evaluate whether the following structure makes sense:

### Equipment
May contain:
- daily operating value
- estimated failure cost
- downtime cost estimate
- asset criticality

### Work Order
May contain:
- estimated cost
- actual cost
- avoided failure cost
- cost justification / decision rationale

### Service Order / Procurement
May contain:
- parts cost
- vendor labor cost
- logistics cost
- ETA / sourcing status

Assess whether this is the right separation of concerns.

---

## 3. How should cost actually be determined?
Critique and improve this financial logic:

### Repair Cost
Repair Cost = parts + labor + logistics

### Failure Cost
Failure Cost = downtime cost + expected damage + disruption cost

### Savings
Savings = failure cost - repair cost

Questions to answer:
- Is this credible enough for operational decisions?
- What assumptions are too weak?
- What data would be required to make this trustworthy?
- What should be estimated vs measured vs learned over time?

### Fleet Superintendent / Technical Superintendent

### Technical Manager / Budget Holder

Assess whether the product can legitimately claim:

AI prediction → work order → service/procurement → completed work → cost captured → failure avoided estimate

## 8. Perform a CRUD Test on the financial system
This is critical.

Test whether the financial model actually works operationally by evaluating Create, Read, Update, Delete behavior across:

- Equipment financial fields
- Work Order financial fields
- Service Order / Procurement financial fields
- Decision / recommendation cost outputs
- ROI / avoided-failure records

### A. CREATE
Can the system cleanly create:
- estimated repair cost
- procurement cost lines
- actual cost records
- avoided failure estimates
- ROI events tied to completed work

Questions:
- Who creates each record?
- At what step in the workflow?
- Is record creation manual, automatic, or hybrid?
- Are there duplicate-entry risks?

### B. READ
Can the right users read the right financial information in the right place?

Questions:
- Can engineers see enough cost context without being overloaded?
- Can procurement see cost lines in their workflow?
- Can management see rolled-up ROI?
- Is the same financial truth visible consistently across Equipment, Work Orders, Procurement, and Analytics?

### C. UPDATE
Can estimates evolve into actuals without breaking trust?

Questions:
- Can estimated repair cost be updated after vendor quote?
- Can actual cost replace or coexist with estimate?
- Can avoided-failure assumptions be revised later?
- Is there versioning or auditability for changed cost assumptions?

### D. DELETE / VOID / SUPERSEDE
This is often ignored. Evaluate it explicitly.

Questions:
- What happens if a work order is cancelled?
- What happens if a service order is duplicated?
- What happens if a vendor quote was wrong?
- What happens if an avoided-failure estimate was based on a bad assumption?
- Should records be hard-deleted, soft-deleted, voided, or superseded?

### CRUD stress questions
Also evaluate:
- Which financial objects are sources of truth?
- Which are derived?
- Which should be immutable after completion?
- Where will reconciliation problems appear?

Do not treat CRUD as a database exercise only — evaluate it as a real operational workflow test.
