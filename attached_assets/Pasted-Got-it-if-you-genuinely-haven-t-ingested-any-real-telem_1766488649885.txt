Got it — if you genuinely haven’t ingested any real telemetry with the legacy collectors yet, then yes: you can remove them now and simplify the system.

Here’s the safest “remove legacy now” approach that still protects you from an accidental “no telemetry at all” situation.

Step 0: Make sure you won’t brick ingestion

Before deleting code, do this on your dev machine (or vessel PC if that’s what you’re using):
	•	Confirm the C# agent is writing to SQLite:
	•	C:\ARUS\data\arus.sqlite exists
	•	raw_frames count increases
	•	Confirm the Node bridge is running (even if Postgres is unreachable):
	•	cursor table exists and the bridge loop stays alive

If either of those isn’t true, deleting legacy will leave you with zero ingestion.

⸻

Option A (recommended): Hard-disable legacy (no code deletions yet)

This is “remove legacy now” operationally, but keeps rollback trivial.
	1.	Set:

	•	ARUS_INGESTION_MODE=hybrid

	2.	In the server bootstrap, only start the bridge. Do not start legacy collectors at all.
	3.	Add a hard guard in legacy collector entrypoints so they can’t run even if imported:

if ((process.env.ARUS_INGESTION_MODE || "hybrid") !== "legacy") {
  throw new Error("Legacy collectors are permanently disabled");
}

This gives you the same outcome (no legacy ingestion), but you can still diff/clean up later.

⸻

Option B: Delete legacy now (what you asked)

Do this in one controlled PR so the repo stays clean.

1) Remove legacy startup wiring

Wherever collectors are started, delete:
	•	startLegacyCollectors()
	•	any imports like server/j1939-collector, server/mqtt-ingestion, direct CAN listeners

Ensure startup only does:
	•	startSqliteBridge()

2) Delete modules

Remove directories/files like:
	•	server/j1939-collector/**
	•	server/mqtt-ingestion/**
	•	any “direct CAN listener” modules

3) Remove dependencies they pulled in

From package.json, remove things only used by legacy ingestion (common ones):
	•	CAN adapter libs
	•	MQTT libs (if only used for ingestion; keep if UI/subscriptions still use it)
	•	serial libs (if only legacy used them; the agent will own serial now)

4) Remove config knobs and docs
	•	Remove legacy env vars and docs for them
	•	Keep only:
	•	agent env vars
	•	bridge env vars
	•	UI status endpoints

5) Ensure idempotency keys are unified

Your bridge idempotency key format should become the standard. Make sure nothing else is generating telemetry rows in parallel.

⸻

One thing you should KEEP (even if deleting legacy)

Keep ARUS_INGESTION_MODE, but set default to hybrid for now. Why?
	•	It gives you a simple safety switch if you later run a “simulator ingestion” path for testing.

You can delete the env var later once everything is stable.

Minimal “proof it’s working” checklist after removal
	•	Agent writes frames: raw_frames grows
	•	Bridge advances cursor: ingest_cursor.last_id increases when Postgres reachable
	•	UI shows:
	•	Agent Connected / Heartbeat
	•	Bridge backlog not exploding
