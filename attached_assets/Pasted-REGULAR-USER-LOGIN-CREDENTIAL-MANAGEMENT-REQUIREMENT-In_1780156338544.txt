REGULAR USER LOGIN CREDENTIAL MANAGEMENT REQUIREMENT

In Admin → Crew Management → Crew Members / Users tab, admins must be able to create and manage login credentials for regular users.

Crew members should have a credential section/form where an admin can:
- assign username
- set temporary password
- require password change on first login
- reset password
- activate/deactivate login access
- see password status
- see last password change date
- see whether password change is required

Important security rule:
Do not display or store plaintext passwords after creation.

Admin should be able to set or reset a password, but should not be able to view the current password later.

The Admin Crew Management page should show credential metadata such as:
- username
- login enabled/disabled
- password last changed
- password change required: yes/no
- last login if available

It should not show:
- current password
- password hash
- reusable secret values

REGULAR USER LOGIN BEHAVIOR

When a regular user logs into the regular User page:
- they authenticate using the username/password assigned from Admin → Crew Management
- if `mustChangePassword` is true, redirect them to a change-password screen before showing the dashboard
- user can change password from their regular User page/account area
- after password change succeeds:
  - password hash is updated server-side
  - `mustChangePassword` becomes false
  - `passwordChangedAt` is updated
  - Admin → Crew Management reflects the updated password status
  - do not expose the new password to admins

REGULAR USER PASSWORD CHANGE UI

On the existing regular User page or user account/settings area, add a password change form.

Fields:
- current password
- new password
- confirm new password

Validation:
- current password is required unless the user is changing a temporary password on first login and the existing auth flow allows that
- new password is required
- confirm password must match new password
- new password must meet minimum strength requirements
- return clear errors for invalid current password or weak password
- never log password values

Suggested password rules:
- minimum 10 characters
- at least one letter
- at least one number
- reject obviously weak passwords if existing validation supports it

AUTH / API REQUIREMENTS

Inspect the existing authentication system first. Reuse the existing auth/session/password hashing approach.

Add or extend APIs as needed.

Suggested endpoints:

POST /api/auth/user-login
Authenticates regular user login credentials and creates a session/token using the existing auth mechanism.

POST /api/me/change-password
Allows logged-in regular user to change their own password.

PATCH /api/admin/crew/:id/credentials
Allows admin to update a crew member’s username, reset password, enable/disable login, or require password change.

GET /api/admin/crew
Should include safe credential metadata:
- username
- loginEnabled
- mustChangePassword
- passwordChangedAt
- lastLoginAt if available

Do not return password hashes.

DATA MODEL REQUIREMENTS

Inspect existing users/auth schema first.

If needed, add or extend fields on the user/crew/auth table:

- username
- password_hash
- login_enabled
- must_change_password
- password_changed_at
- last_login_at

If the app already has an auth users table, prefer linking crew members to the existing auth user record rather than duplicating credentials.

Password storage:
- hash passwords with the existing app’s password hashing system
- if no hashing system exists, use a secure password hashing library already present in the repo
- never store plaintext passwords
- never return password hashes to the frontend

AUTHORIZATION REQUIREMENTS

- only admins can create/reset regular user credentials from Crew Management
- regular users can only change their own password
- regular users cannot update another user’s credentials
- admin credential APIs must be server-side role-gated
- disabled users/loginDisabled accounts cannot log in
- inactive crew members should not be able to log in unless existing business rules allow it

CREW MANAGEMENT UI REQUIREMENTS

In Crew Management → Crew Members tab, add a credentials section inside the user assignment/edit dialog.

Admin can:
- enter or edit username
- set temporary password
- require password change on next login
- reset password
- enable/disable login
- view password status metadata

Display:
- username
- login enabled status
- password change required status
- password last changed date
- last login date if available

Do not display current password after save.

REGULAR USER PAGE UPDATE

When a regular user changes their password:
- show success toast/message
- invalidate/refetch relevant user/account state
- Admin Crew Management should show updated password status on next load/refetch

If the user is required to change password:
- show change-password form before dashboard content
- after successful change, show the role-based User dashboard

TYPESCRIPT SAFETY

Add typed schemas for:
- admin credential update payload
- user password change payload
- regular user login payload
- safe credential metadata response

No `any`.
No `as any`.
No unsafe double-casts.
Do not type password fields loosely.

Suggested shared types:
