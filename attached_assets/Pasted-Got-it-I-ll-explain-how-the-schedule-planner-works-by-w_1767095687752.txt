3️⃣ Center Timeline – Assignment Blocks (Core Interaction Surface)

What you see
	•	Horizontal bars with:
	•	crew name
	•	role
	•	color coding
	•	Blocks span across days
	•	Some blocks show warning icons (e.g. fatigue)

How it works
Each block is a single Assignment:

Assignment {
  crewId
  vesselId
  role
  startDate
  endDate
  status (draft/published)
}

Block behaviors
	•	Drag block → changes vessel or dates
	•	Resize block edges → adjusts start/end
	•	Hover → shows quick warnings
	•	Click → opens Assignment Drawer

Visual rules
	•	Color = role or rank (existing design system)
	•	Icons:
	•	🔴 red = HARD constraint violation
	•	🟠 amber = SOFT warning
	•	Draft assignments look slightly muted

👉 Replit rule:
Blocks are optimistic UI: move first, validate second.

4️⃣ Constraint Warning Tooltip (Inline Feedback)

What you see
	•	Tooltip: “Fatigue Risk – Exceeds max onboard days (28 days)”

How it works
	•	On every create/move/resize:
	1.	Planner calls checkConstraints(draftAssignment)
	2.	Returns structured results
	3.	UI overlays icons + tooltips

Important
	•	HARD ≠ allowed to publish
	•	SOFT ≠ blocked, just warned

👉 Replit rule:
Planner never decides rules — it only displays domain results.

7️⃣ Apply Changes Button (Write Boundary)

What it does
	•	Commits the draft assignment changes.

Under the hood
	1.	UI calls SchedulingService.upsertAssignment()
	2.	Assignment is saved locally
	3.	Pending sync op is created
	4.	Constraint state is stored
	5.	Notifications are NOT sent yet (unless auto-publish)

👉 Replit rule:
This is the only write entry point from the planner.

8️⃣ Publish vs Draft (Lifecycle Control)

Not shown explicitly, but implied

Draft
	•	Visible to admins
	•	No emails sent
	•	Can violate SOFT rules

Publish
	•	Validates zero HARD violations
	•	Marks assignments as Published
	•	Triggers notification events:
	•	Crew / Admin / Both (from settings)
	•	Uses existing email system

👉 Replit rule:
Publish = validation + notifications + audit.
