PHASE 2 – Design a Versioned Export/Import Strategy



Based on the analysis, design an export/import system with these properties:

Logical, not engine-bound

Export is based on logical entities (org, vessels, equipment, work orders, etc.), not the physical DB engine.

The same export format must work whether data is currently in SQLite only or synced to Turso/libSQL/Postgres.

Versioned manifest

Each export must include a manifest object, e.g.:

{
  "exportVersion": 1,
  "appVersion": "<current app version if available>",
  "schemaVersion": "<current schema version or migration tag>",
  "exportedAt": "ISO_TIMESTAMP",
  "orgId": "default-org-id",
  "entities": {
    "vessels": { "count": 12 },
    "equipment": { "count": 180 },
    "workOrders": { "count": 57 },
    "telemetry": { "count": 102934 }
  }
}


This manifest is used during import to handle compatibility and optional transforms.

Format & scope

Prefer a single archive structure, e.g.:

manifest.json

vessels.jsonl

equipment.jsonl

work_orders.jsonl

telemetry.jsonl (or chunked if huge)

Use JSONL (newline-delimited JSON) per entity for streaming.

Support at minimum:

Export by org (all data for an org)

Optionally export subset (e.g. by vessel) if easy and safe.

Security and tenancy

Export must respect x-org-id and only dump data for that org.

Never bypass tenant isolation middleware.

Offline-first & multi-device

Design so exports from embedded/vessel SQLite can be imported into:

Another desktop device

A “newer” build of the app with migrated schema

If relevant, allow import into the cloud/libSQL/Turso backend (via an admin script) for centralization.



Output a short design section in Markdown before coding:

Any necessary migration/versioning bits
