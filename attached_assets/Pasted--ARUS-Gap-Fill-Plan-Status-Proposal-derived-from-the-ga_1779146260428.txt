## 0. Reality calibration vs. the gap doc

Before sequencing the work, a few corrections to ground the plan in the actual repo state:

**Already in place (the gap doc undercounts these):**

- `pg-boss` is in `package.json` dependencies — the background-job framework is installed but `server/background-jobs.ts` is an explicit stub (`// Stub file - background jobs consolidated`). So this is *wire-up*, not adopt-from-scratch.
- `server/observability/` is rich: ~23 metric modules (alerts, ML, MQTT, optimizer, scheduler, RAG, security, telemetry-ingestion, http, websocket, hub-sync, etc.) all on `prom-client`. **Metrics tier is solid.** What’s missing is *tracing* (OTel) and *log aggregation*, not metrics.
- `server/swagger/` has a working OpenAPI spec generator with paths for equipment, vessels, work orders, PdM, telemetry, health, misc. Swagger UI exists. Versioning prefix is the gap, not OpenAPI itself.
- `server/infrastructure/feature-flags.ts` exists (270 lines) but is env-var-only and scoped to the tenant-repo migration. The gap is generalizing it to remote/per-tenant/per-user, not building a flag system from zero.
- `server/domains/` has 30+ domain folders — hexagonal structure is largely realized. The decommission target is the legacy `server/repositories.ts` proxy facade (144 lines, still re-exporting `db*Storage` singletons), not the whole hexagonal migration.
- `server/backup-recovery/` exists as a module. Backup *exists*; PITR + restore drill + cross-region replica is the gap.
- Edge agent `apps/agent/Arus.TelemetryAgent/` has J1939 CAN, Serial, and Simulated sources. **No OPC UA, NMEA 2000, or Modbus** on the edge (Modbus exists server-side for the Aquametro FMCC integration only).

**Confirmed absent (gap doc is right):**

- i18n: zero references to `i18next`, `useTranslation`, or `FormattedMessage` anywhere in `client/`. `<html lang="en">` is hard-coded.
- Sentry: not in dependencies. OpenTelemetry: not in dependencies.
- SAML/OIDC/SCIM: zero references.
- Storybook: no `.storybook/`, no stories.
- IaC: no `.tf`, no Pulumi, no Crossplane.
- Visual regression: not in CI.
- Design tokens: Tailwind config is present but there’s no token system or theming layer beyond it.

**Net effect on the plan:** Several “build this from scratch” items in the gap doc are actually “finish wiring what’s there.” That trims roughly 3–4 weeks of effort off the original re-estimate.

-----

## 1. Sequencing principle

The waves below are ordered by three forces:

1. **Retrofit cost compounds.** i18n, design tokens, API versioning, OTel context propagation, and feature-flag generalization all get exponentially more expensive the more code lands on top of them. These go in Wave 0 even though they don’t directly unlock revenue.
1. **Enterprise procurement has long lead times.** SOC 2 Type II takes 6–12 months end-to-end (3 months policies + 3–6 month observation window + audit). If there’s an enterprise pipeline at all, the readiness program starts in Wave 1 or it doesn’t land in time.
1. **Operational reliability gates the rest.** OTel + Sentry + DR drills before scaling customer count, or debugging incidents becomes guesswork.

Mobile, edge expansion, and ecosystem connectors come after, because they’re additive — they don’t get more expensive by being delayed (they just don’t ship).

## 3. Three highest-leverage items (matching the gap doc’s call)

1. **SOC 2 readiness kickoff (Wave 1.7)** — the 6-month observation window is the long pole. Start now or it doesn’t land in time for enterprise deals. Cost: ~$30–80k tooling + auditor; ~0.3 FTE coordination.
1. **SSO/SAML + SCIM (Wave 1.1 + 1.2)** — every enterprise IT department asks for this on day one of the security review. Two weeks of work; gates millions of dollars of pipeline.
1. **OpenTelemetry + Sentry (Wave 0.4 + Wave 2.1)** — the moment paying customers report bugs you can’t reproduce, debugging without these is guesswork. Sentry can land in a day in Wave 0; full OTel comes in Wave 2.

## 4. What to explicitly skip or defer

- **LaunchDarkly / Unleash / OpenFeature SDK.** The generalized in-house FeatureFlagManager (Wave 0.6) is enough until you have 50+ flags and need experimentation tooling. Don’t pay for the SDK yet.
- **MLflow.** The `mlModels` schema + a registry service in TypeScript is enough until you have 50+ models. MLflow is operational overhead now, not yet a saving.
- **GPU / Triton inference serving.** ONNX-on-CPU and the small models you’d actually run on edge or in cloud do not need this. Revisit if a customer asks for vision models on engine room cameras.
- **Public developer portal as a P0.** Defer until 3+ partners have asked for the same integration. The internal Swagger UI is enough until then.
- **Pact contract testing.** You’re a monorepo with a single FE consumer of the API. Pact pays off when you have multiple independent consumers. Defer.
- **Schema registry (Avro/Protobuf).** Wait until a real streaming spine (Redpanda/NATS) is in. Otherwise you’re choosing the encoding without the system that uses it.
- **Mutation testing.** Useful but expensive in CI minutes. Defer to after Wave 6.
- **Capacitor → React Native migration.** Capacitor is fine for the maritime use case. Don’t get drawn into a port.

```
Months 1-2:   Wave 0 (platform foundations) + Wave 1 kickoff (SSO, SCIM, MFA, SOC 2 start)
Months 3-4:   Wave 2 (observability + reliability) + Wave 1 wrap
Months 5-6:   Wave 3 (ML ops) + Wave 4 start (edge)
```

## 6. Decision points

A few things in this plan depend on choices that aren’t mine to make. Flagging them so they get decided early, not in the middle of a wave:

1. **Self-host vs. SaaS for observability.** Tempo + Loki + Grafana (self-host) vs. Datadog or Honeycomb (SaaS). Affects Waves 0.4, 2.1, 2.2 setup. Cost vs. ops trade-off.
1. **Hosted Postgres provider.** Neon, RDS, Cloud SQL, or Supabase. Drives PITR + cross-region replica choices in Wave 2.3/2.4 and IaC in Wave 0.7.
1. **SOC 2 tooling.** Vanta, Drata, or Secureframe. All three work; pick one in Wave 1.7 and don’t switch.
1. **Auth IdP for SSO testing.** Customer expectations will drive priority order (Okta vs. Azure AD vs. Google Workspace). Want at least one customer-validated by end of Wave 1.
1. **Edge OTA channel topology.** Per-vessel S3 bucket vs. central manifest server. Sat-link bandwidth and air-gapped deployments push toward staged manifest mirroring; cloud-only fleets are fine with a central server.

## 7. What success looks like at each wave

- **End of Wave 0:** No new code lands without i18n keys, no new endpoint without `/v1/` prefix, every page error is captured in Sentry, no `OPENAI_API_KEY` in `.env`.
- **End of Wave 1:** A customer with Okta can log in via SAML; SCIM provisions/deprovisions their users; SOC 2 auditor has scoped the engagement.
- **End of Wave 2:** An incident can be traced from a clicked button in the cockpit, through the API, through a pg-boss job, to an MQTT message on the agent. Restore drills happen quarterly. EU tenants run from EU read replicas.
- **End of Wave 3:** Every production model has drift dashboards. Copilot eval runs on every PR. OpenAI spend is bounded per tenant.
- **End of Wave 4:** A vessel with OPC UA + NMEA 2000 + Modbus equipment integrates without custom code. Agents update themselves over sat link. Sub-second alarms work during cloud outage.
- **End of Wave 5:** Bridge officers use the iPad app. Push notifications wake the app. Cockpit redesign is visually regression-tested on every PR.
- **End of Wave 6:** `server/repositories.ts` is deleted. Customers pull their data into Snowflake. Tenants can be hard-deleted on request.
- **End of Wave 7:** Two of (SAP PM / Maximo / Wärtsilä FOS / ABB Ability) work. Public API has its first external user.
