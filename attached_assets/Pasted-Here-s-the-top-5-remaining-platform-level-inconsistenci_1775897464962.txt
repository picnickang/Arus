Here’s the top 5 remaining platform-level inconsistencies, ranked by long-term risk first, then by how worthwhile they are to clean up next.

1. Legacy storage facade is still a live backend pattern

This is still the biggest convergence issue. server/storage.ts is still a central singleton/facade with initialization logic, exported types, and broad usage expectations, even though newer parts of the platform are moving toward domain-oriented routing and adapters. That means ARUS still has more than one valid answer to “where should backend data access go?”

Why it matters: this shapes every new backend feature. If you do not lock this down, future work will keep splitting between facade-style access and domain/port-oriented access.

What I’d do next: ban new imports from server/storage.ts outside an explicit legacy allowlist, and force new code through one approved domain/repository pattern.

2. Route registration is cleaner, but not yet singular

server/routes.ts is clearly trying to become a thin coordinator, and it uses registerAllDomainRouters, which is good. But that same file still imports older route modules and legacy route categories, and the codebase still carries legacy wrapper registration around newer domain routers. That means backend route wiring has a canonical future, but not a single present.  ￼

Why it matters: this is the backend equivalent of having two front doors. It confuses contributors and makes middleware/application policy harder to reason about over time.

What I’d do next: require all new feature routes to register only through the domain router registry, and move any non-exempt legacy route mounts onto a retirement list with dates.

3. Frontend information architecture still carries a lot of migration residue

The frontend is more organized than before, but legacy-redirects.ts still contains a large migration map, and navigationConfig.ts still includes explicit route migration helpers plus legacy support for older homepage grouping formats. That means the product’s navigation model is better, but the old map is still being carried around in a fairly direct way.

Why it matters: this increases surface area, keeps old mental models alive, and makes it easier for future UI work to keep “supporting the old shape” instead of finishing the new one.

What I’d do next: rank redirects by actual traffic or importance, then retire them in batches instead of letting them remain indefinitely.

4. Dual-database support is disciplined, but still a permanent complexity multiplier

This is not sloppy, but it is still a major source of platform-wide inconsistency pressure. shared/schema-runtime.ts explicitly switches runtime exports between PostgreSQL and SQLite/Vessel schemas, and it does so across a very broad set of tables. There is also a backward-compatible SQLite vessel schema re-export that is itself marked deprecated for new code.

Why it matters: every schema evolution, every new table, and every edge-case behavior has a higher maintenance cost. This is probably worth it for vessel/offline operations, but it remains one of the largest structural taxes in the system.

What I’d do next: keep the runtime switch, but add stricter CI checks that verify every new schema/table change is implemented in both modes or explicitly exempted.

5. Legacy product overlap still exists in PdM and sync/admin flows

There is still evidence of transitional overlap in user-facing product areas. The codebase still carries PdM migration notes and replacement mappings that explicitly say legacy pages/hooks/routes should be deprecated rather than removed yet, and SyncAdmin.tsx still queries a legacy sync health endpoint alongside the newer sync status path. That means some product areas still have “new plus old” living together.

Why it matters: this is where architectural drift turns into user drift. It makes it easier to preserve duplicate product paths and harder to tell which one is truly authoritative.

What I’d do next: pick the next domain after work orders, probably PdM or sync/admin, and apply the same pattern you just used: one path, one contract, no fallback, old path removed.

The order I’d tackle them

If you want the highest leverage sequence, I’d do it like this:

1. storage facade retirement
2. route registration enforcement
3. PdM/sync overlap cleanup
4. redirect retirement
5. stronger dual-db guardrails

That order gives you the most future benefit, because the first two affect how every new feature gets built.

The simplest executive summary

ARUS is no longer suffering from one glaring broken example. Now the remaining issues are platform convergence issues:
	•	backend access still has more than one valid pattern
	•	route registration is still partly transitional
	•	frontend navigation still carries meaningful migration baggage
	•	dual-db support remains costly but intentional
	•	a few domains still have “new + legacy” product overlap

That means the platform is in a better place now: the next work is not rescue work, it is convergence work.
