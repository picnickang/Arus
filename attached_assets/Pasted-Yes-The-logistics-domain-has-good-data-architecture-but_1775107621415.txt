Yes. The logistics domain has good data architecture but the UX has gaps that would frustrate a real purchasing officer or chief engineer using this daily. Here’s what I’d change, in order of impact:
1. The purchasing workflow is split across too many pages with no visual pipeline.
A purchase request starts in the work order detail drawer (WorkOrderRequestsTab), gets created via a dialog, then the user has to navigate to /purchase-requests to see it, then to /purchase-orders to see the PO that was generated from it, then back to /service-orders if it involved external service. There’s no single view that shows “here’s where this request is in the pipeline.”
I’d add a status pipeline strip at the top of the PR detail page: Request Created → Sent to Supplier → Quote Received → PO Issued → Shipped → Received → Fulfilled. Each step is tappable and shows the relevant data. The user sees the entire lifecycle without navigating between 4 pages. The data for this already exists in the purchaseRequestEvents and purchaseOrderEvents tables — it just needs to be assembled into a timeline.
2. The inventory page doesn’t tell you what to do.
The stat cards show “12 Critical/Out” and “8 Low Stock” — but when you click them, they open the LowStockReplenishmentPanel which is a separate sheet. The main table doesn’t change. A purchasing officer’s workflow is: see what’s low → check which parts are needed for upcoming work orders → create a PR for the right quantities → send to supplier.
The replenishment panel should pre-populate with parts sorted by urgency (critical first, then low stock), show the upcoming work orders that need each part, and let the officer select multiple parts and create a single multi-line PR with one click. The suggestPartsForSensorIssue and checkMultiplePartsAvailability APIs exist — they’re just not surfaced in the replenishment flow.
3. The service order calendar exists but isn’t wired up.
ServiceOrderCalendar.tsx is a complete, well-built component sitting unused. Service orders are inherently time-based (scheduled start, scheduled end, downtime windows) but the ServiceOrdersPage shows them as cards in a grid. A fleet manager planning maintenance windows needs to see “what service work is happening this month across the fleet” — that’s a calendar view.
Add a toggle on the ServiceOrdersPage header: “Cards | Calendar”. Default to cards for the list/filter workflow, but let users switch to calendar for scheduling visibility. The component is already built and accepts the exact data shape the page has.
4. There’s no “reorder from last PO” shortcut.
When a part runs low, the most common action is to reorder from the same supplier at the same price as last time. Currently this requires: open part detail → check supplier → navigate to purchase requests → create new PR → add items → set quantities → select supplier. That’s 6 steps for what should be 1 tap.
Add a “Reorder” button on each critical/low-stock row in the inventory table that pre-fills a PR with the same supplier, same quantity (based on max stock minus current stock), and same unit cost as the last PO for that part. The lastOrderDate, primarySupplierId, and standardCost fields on the parts table have the data — it just needs a shortcut action.
5. The work order → service request → service order flow has too many dialogs.
Creating a service order from a work order currently opens EnhancedServiceRequestDialog — a 300-line dialog with 15+ fields including MOC number, certificate renewals, severity badges, assistance tags, probable cause, and downtime windows. For a chief engineer who just needs to say “I need a technician to fix the pump next Tuesday,” this is overwhelming.
I’d split it into two modes: “Quick Request” (provider + equipment + description + date — 4 fields) and “Detailed Request” (the current full form). Default to Quick Request with a “Show advanced options” toggle that reveals the rest. Most service requests don’t need MOC numbers or certificate renewals — those are for drydock-level work.
6. Supplier performance data isn’t visible where decisions are made.
The evaluateSupplierPerformance function computes on-time delivery rate, quality rating, and performance score — but this data only appears in the beast-mode API. When a purchasing officer is selecting a supplier for a new PO, they see a dropdown with supplier names. They don’t see “Supplier A: 95% on-time, 8.5/10 quality” vs “Supplier B: 72% on-time, 6.1/10 quality.”
Surface the supplier performance score inline in every supplier selection dropdown. The SupplierCard already shows these metrics — extract a SupplierSelectOption component that shows name + performance badge and use it in the PR form, PO form, and service order form.
7. No batch actions on the inventory table.
The VirtualizedInventoryTable supports row selection (selectedItems set) but there are no batch actions. A purchasing officer reviewing inventory doesn’t want to create individual PRs for each low-stock part — they want to select 8 parts, click “Create PR for selected,” and get a single multi-line purchase request.
Add a floating action bar that appears when items are selected: “X items selected | Create PR | Export Selected | Mark Inactive”. The multi-part PR creation is already supported by MultiLinePartsRequestDialog — it just needs the selected items passed as initial data.
8. The SQLite schema for inventory doesn’t match the PostgreSQL schema.
The sqlite-schema/inventory.ts defines partsInventorySqlite with fields like reorderLevel, reorderQuantity, while the PostgreSQL schema uses minStockQty, maxStockQty. The stockSqlite table has quantity (integer) while the PostgreSQL stock has quantityOnHand (real). When the Tauri desktop app syncs offline data to the server, these field mismatches will cause silent data loss or sync failures. This isn’t a UX issue per se, but it’ll manifest as “I updated the stock on the vessel but it didn’t show up in the office” — which is a UX failure.
The schemas need to be aligned, or the sync layer needs explicit field mapping. Given that the SQLite schema appears to be for offline vessel deployments and the PostgreSQL is the cloud schema, this is critical for the offline-first workflow you’ve designed.
The one change I’d make first: The reorder shortcut (#4). It takes 15 minutes of implementation — a button on the inventory row that calls the existing createPurchaseRequest mutation with pre-filled data from the part’s last PO. It eliminates the most repetitive workflow in maritime inventory management.​​​​​​​​​​​​​​​​
