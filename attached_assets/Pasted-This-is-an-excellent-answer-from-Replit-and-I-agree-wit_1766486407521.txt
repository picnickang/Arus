Final Architecture (Lock This In)

Electron / Node (existing ARUS app)
	•	UI
	•	SQLite (Turso sync, schema unchanged)
	•	MQTT ingestion (unchanged)
	•	Batch writer, outbox, sync, ML, alerts (unchanged)

NEW: C# Windows Service (Telemetry Agent)
	•	Hardware I/O only:
	•	CAN / J1939
	•	J1708 / J1587
	•	Serial
	•	Writes raw frames only into SQLite
	•	Exposes gRPC over localhost
	•	Zero business logic

Think of the C# service as a driver shim, not an app.

Phase 1 — C# Windows Service Skeleton (Day 1)

Phase 2 — IPC Contract (gRPC) (Day 1–2)

Goal: Stable boundary between worlds.

gRPC API (minimal)

service TelemetryAgent {
  rpc GetStatus(Empty) returns (AgentStatus);
  rpc StartSource(SourceConfig) returns (Ack);
  rpc StopSource(SourceId) returns (Ack);
  rpc SubscribeStats(Empty) returns (stream IngestStats);
}

Important rule
	•	❌ NO raw telemetry streaming to Electron
	•	✅ Only health + ingest stats

Deliverable
	•	Node client connects via localhost gRPC
	•	UI shows live agent stats

⸻

Phase 3 — SQLite Write-Through Layer (Day 2–3)

Goal: Agent writes raw frames into your existing DB.

Tasks
	•	Use same SQLite file path as Node app
	•	Enable WAL mode
	•	Implement table:
	•	raw_frames (append-only)
	•	Write strategy:
	•	In-memory queue
	•	Flush in batches
	•	Backpressure when disk slow

Rules
	•	❌ No decoding
	•	❌ No feature extraction
	•	❌ No alerts
	•	✅ Raw frames only

Deliverable
	•	Kill UI → agent still writes
	•	Reboot → no data loss

⸻

Phase 4 — Hardware Ingestion Plugins (Day 3–5)

Goal: Real devices, zero UI involvement.

Interfaces

interface IIngestSource {
  Start();
  Stop();
  Status();
}

Implement
	•	Serial (RS232/485)
	•	CAN/J1939 (adapter SDK)
	•	J1708/J1587
	•	Reconnect + heartbeat
	•	Timestamp normalization

Deliverable
	•	Plug device → data appears in SQLite
	•	Unplug → auto-reconnect
