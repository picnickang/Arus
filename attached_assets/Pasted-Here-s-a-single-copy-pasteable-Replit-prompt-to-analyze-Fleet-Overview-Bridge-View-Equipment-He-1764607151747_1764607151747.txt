Replit prompt:



You are an expert React + UX engineering assistant working on a maritime PdM Electron/web app (ARUS). The app has multiple overview/monitoring tabs that feel redundant and cluttered:

Fleet Overview

Bridge View

Equipment Health



I want you to analyze these screens, find overlaps, and propose/implement a simplification so the UI feels more coherent and less cluttered — without breaking existing behavior.

0. PROJECT CONTEXT (VERY IMPORTANT)



The stack you should assume (do NOT replace, only extend/refine):

Frontend: React + TypeScript, shadcn/ui + Tailwind, TanStack React Query, React Router (or similar).

Desktop: Electron wrapper (but same React app runs in browser).

Backend: Node/Express, PostgreSQL + SQLite/libSQL (dual storage), MQTT-based telemetry, PdM analytics, etc.

Domain: Marine predictive maintenance, fleet monitoring, equipment health, logbooks, crew management.



Use existing conventions (components, hooks, styling) wherever possible.

1. DISCOVER & MAP THE THREE TABS



First, scan the codebase to find all components/pages related to:

FleetOverview

BridgeView

EquipmentHealth

Or similarly named routes (e.g. /fleet, /bridge, /equipment/health, etc.)



For each of these three areas:

Identify:

The main React component file(s).

Any shared hooks (useFleetOverview, useEquipmentHealth, etc.).

Any shared layout components (dashboards, cards, tabs, charts).

For each tab/page, describe in comments (at the top of the main component file):

What data it shows (fleet-level, vessel-level, equipment-level).

Main widgets (maps, KPIs, charts, tables).

Primary user flows (“monitor fleet status at a glance”, “see live telemetry per vessel”, “drill into equipment health”, etc.).

Any unique features that do not appear in the other two views.



Do NOT refactor yet. Just map and annotate.

2. ANALYZE OVERLAP & REDUNDANCY



Next, still at the “analysis/planning” stage:

Create an internal summary (in comments or a new docs/ui-overview.md file) comparing:

Fleet Overview vs Bridge View vs Equipment Health:

Which metrics are shared (e.g. RUL, alarms, status, KPIs, telemetries)?

Which visualizations are similar (e.g. cards, tables, charts, maps)?

Which actions overlap (acknowledge alert, open vessel detail, open equipment detail)?

Identify:

UI patterns that are duplicated (same card/table layout but slightly different labels).

Any conflicting terminology or inconsistent filters.

Components that can be generalized (e.g. StatusCard, HealthKpiGrid, FleetMapSection) and reused across the three views.

Propose (in comments or doc) a simpler IA (information architecture) based on these principles:

One primary overview for the whole fleet (high-level).

One vessel-level dashboard (bridge-style) that shows:

Live telemetry + route/location view for a selected vessel.

One equipment health view where:

Filtering is clear: by vessel, by system, by severity.

Avoid creating multiple screens with almost identical cards/tables; prefer reusable sections with different filter presets.



Do not change code yet; just write the proposal and clearly mark it as a plan.

3. DESIGN A CONSOLIDATED LAYOUT (UX PLAN)



Now design the target UX as a plan, using existing shadcn/Tailwind patterns:

Fleet Overview (top level):

Simple 3-column KPI header for fleet:

vessels operational / in maintenance / in alert.

Overall alert count by severity.

High-level fuel/emissions summary (if available).

A fleet list/table or map where each row/card links into the vessel dashboard (bridge-style view).

Lean visuals: avoid duplicating detailed equipment cards here.

Vessel Dashboard (“Bridge View”):

Keep per-vessel layout, but:

Top: vessel identity + status (RUL summary, alerts, ETA, location).

Left side: live telemetry & health summary (can reuse components from equipment health but filtered by vessel).

Right side: route/location widget (if available) plus key alerts list.

This is where the master for live operational monitoring lives.

Equipment Health:

A filter-first design:

Filter by vessel, system, severity, status.

Use a table/list or card grid for equipment, but re-use the same card pattern as in other views.

Provide drill-down into equipment detail (if already exists).



Record this UX plan as comments first and ensure:

It describes how existing components can be refactored/reused.

It avoids adding brand-new visual systems unnecessarily.

4. IMPLEMENTATION – STEPWISE REFACTOR (NO BIG-BANG BREAKAGE)



Now start implementing, carefully and incrementally.



4.1 Extract Shared Components

From Fleet Overview, Bridge View, and Equipment Health, identify repeating chunks like:

Status KPI cards.

Alert summary blocks.

Equipment health cards.

Basic vessel summary row/card.

Extract these into shared components in a sensible folder (e.g. src/components/fleet/ or similar), such as:

FleetKpiHeader

VesselStatusCard

EquipmentHealthCard

AlertSummaryPanel

Replace duplicated inline JSX in each tab with these shared components.

Ensure props are generic, e.g.:

status, severityCounts, vessel, equipment, rather than hard-coded to a specific view.

Keep typings strict (TypeScript).

Confirm that after this extraction, behavior and data remain the same as before.



4.2 Simplify Each Tab’s Layout

Apply the UX plan from Step 3:

Fleet Overview:

Ensure it focuses on high-level, fleet-wide info:

Use shared KPI header and simple fleet list.

Remove or demote detailed equipment-level widgets that are better suited to Equipment Health or Vessel Dashboard.

If needed, keep “View in Bridge View” or “View Equipment Health” actions.

Bridge View / Vessel Dashboard:

Convert it into the main per-vessel dashboard:

Use shared components to show live telemetry + health summary for the selected vessel.

Avoid duplicating fleet-level KPIs.

Ensure navigation from Fleet Overview → Bridge View is clean and consistent.

Equipment Health:

Ensure filters are clear and consistent with the rest of the app.

Reuse the equipment cards/rows used elsewhere.

Avoid showing vessel-level KPIs that belong to Fleet Overview.

Where you simplify or remove UI blocks:

Add comments explaining what was removed, why, and where similar information can be found now (to help future devs).

5. NAVIGATION, ROUTING, AND STATE CONSISTENCY

Ensure that routing and navigation are consistent:

From Fleet Overview → clicking a vessel row opens that vessel in Bridge/Vessel Dashboard.

From Equipment Health → clicking an equipment entry can:

Show equipment detail, and/or

Optionally open the vessel’s dashboard with that equipment highlighted.

Check shared state:

If there is a global selectedVessel or organizationId, ensure the tabs share it consistently.

Avoid duplicating “current vessel” logic in three different places.

Update breadcrumbs / page titles to be clear:

“Fleet Overview”

“Vessel Dashboard – [VESSEL NAME]”

“Equipment Health”

6. VISUAL CLEAN-UP & CLUTTER REDUCTION



Do a final pass to reduce clutter:

Remove duplicated legends, repeated section titles, and redundant labels.

Standardize spacing, typography, and icon use across these three views using existing shadcn/Tailwind tokens:

Use consistent card spacing p-4, heading text-xl font-semibold, etc.

Keep icon sizes consistent (e.g. h-4 w-4 or h-5 w-5 across cards).

Ensure responsiveness:

Desktop: multi-column layout.

Smaller screens: stacked layout with sane scroll behavior.

7. SAFETY CHECKS & TESTS

Make sure all changes are backwards compatible:

No breaking changes to APIs or backend.

No removal of critical content, only relocation/simplification.

Run the test suite (if present). If none exist for these views:

Add light tests to confirm:

Each tab renders without crashing.

Shared components render given representative data.

Basic navigation between Fleet → Bridge → Equipment Health works.

Fix any TypeScript errors, React key issues, or React Query warnings introduced by the refactor.
