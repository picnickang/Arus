Assumptions: this is a static source review after extracting the tarball; I did not have prod logs, runtime metrics, a live DB, model artifacts, or `node_modules`. I assume ARUS is intended to run as a real multi-tenant SaaS with `REQUIRE_TENANT_AUTH=true`, not only legacy `default-org-id` mode.

**Executive Summary**
- The biggest risk is tenant isolation: RAG messages, default-org fallbacks, direct `x-org-id` reads, and WebSocket system broadcasts create credible cross-tenant leak paths.
- The DDD/hexagonal architecture is aspirational but not enforced yet: guardrails report 601 domain/storage leaks and boundary violations.
- Offline sync is not production-safe: PostgreSQL/SQLite drift is failing, conflict resolution is a no-op shim, and key workflows are not queueable offline.
- CI is currently red by construction: multiple guard scripts fail, test scripts target missing `tests/*` paths, and there is no `.github/workflows` directory.
- ML lifecycle is partly stubbed: training defaults to an in-process stub runner, and inference falls back to heuristics without making that state operationally visible.
- The product has strong workflow foundations: guided work-order closeout, smart replenishment, PDM scheduling, RLS design, structured logging, route code splitting, and guard scripts are all good starts.
- The highest-return fixes are tenant checks, schema/migration cleanup, CI repair, WebSocket scoping, and making PDM schedule actions actually writable.
- Operationally, users still hit dead ends: “move task” is disabled, closeout is mostly free text, PR creation can leave partial drafts, and crew scheduling reports gaps without resolving them.

**Detailed Findings**

**1. Architecture**
**A1. Domain boundaries are not enforced**  
Category: Architecture & boundaries. Severity: High. Evidence: `scripts/check-domain-boundaries.mjs` fails on `server/domains/me-portal/me-portal-service.ts:12`, `system-admin/routes/settings-routes.ts:15`, `tenant-routes.ts:24`; `check-domain-leaks` reports 601 leaks. Why it matters: domain changes can cascade through route/storage imports. Recommendation: move cross-domain calls behind ports/application services and make boundary checks blocking again. Effort: L. Blast radius: broad, but mostly import/interface cleanup.

**A2. Route/storage coupling bypasses application contracts**  
Category: Architecture & boundaries. Severity: High. Evidence: direct `db`/storage imports in domain routes like `server/domains/sensors/routes.ts:3`, `equipment/routes.ts:6`, `work-orders/application/work-order-service.ts:9`. Why: validation, tenant scoping, events, and audit behavior become inconsistent per route. Recommendation: standardize repositories behind domain ports. Effort: L. Blast radius: moderate across route handlers.

**2. Data Model**
**D1. Dual schema drift is actively failing**  
Category: Data model. Severity: High. Evidence: `node scripts/validate-dual-schema.mjs` reports 15 new drifts, including `workOrders.workOrderType`, `portCall`, `drydockWindow`, `requestIdempotency`, `sheetLock`, and `scheduleAssignments`. Why: offline clients can write shapes the cloud cannot accept, or lose fields during sync. Recommendation: hand-write paired PG/SQLite migrations and add sync-contract tests. Effort: M. Blast radius: offline/mobile/Tauri sync.

**D2. Tenant child tables lack `org_id`/RLS**  
Category: Data model. Severity: High. Evidence: `shared/schema/rag.ts:157` `ragMessages` has only `conversationId`; `kbDocVersions` lacks `orgId` at `shared/schema/rag.ts:86`; `migrations/0018_rls_policies.sql` explicitly omits several tables. Why: every query must remember parent joins perfectly. Recommendation: add `org_id` to tenant child tables or enforce scoped repository APIs and DB views. Effort: M. Blast radius: data migration/backfill.

**D3. Migration workflow is split and brittle**  
Category: Data model. Severity: High. Evidence: `drizzle.config.ts:31` references missing `./shared/schema/email-templates.ts`; `scripts/run-sql-migrations.mjs:9` says Drizzle tracks only 5 of 23 SQL migrations; `package.json:35` still uses `server/scripts/migrate.ts`. Why: one deploy path can skip migrations another applies. Recommendation: make one migration runner authoritative and fix schema list. Effort: S/M. Blast radius: deploy tooling.

**3. Security**
**S1. RAG messages can be read cross-tenant by guessed conversation ID**  
Category: Security. Severity: Critical. Evidence: `server/routes/rag-routes.ts:206` returns `getMessages(id)` without org check; `conversation-service.ts:154` filters only by `conversationId`; `ragMessages` has no `orgId` at `shared/schema/rag.ts:157`. Why: sensitive copilot/chat content can leak. Recommendation: require `getConversation(id, orgId)` before messages, add `orgId` to messages, and add regression tests. Effort: S/M. Blast radius: RAG routes/schema.

**S2. Tenant context still falls back to client-controlled/default orgs**  
Category: Security. Severity: High. Evidence: `server/routes/rag-routes.ts:148`, `:167`, `:188` hardcode `DEFAULT_ORG_ID`; `sensor-management/routes/telemetry-routes.ts:17` reads `x-org-id`; `check-org-context-boundary` fails. Why: tenant isolation depends on caller behavior. Recommendation: source org only from authenticated `req.orgId`, fail closed in tenant mode. Effort: S/M. Blast radius: legacy single-tenant paths.

**S3. Copilot client bypasses shared Authorization headers**  
Category: Security. Severity: High. Evidence: `AgentChatPanel/index.tsx:360` XHR sets only `x-org-id`; streaming headers at `:405` also omit `Authorization`; server requires bearer auth in `server/security/authentication.ts:35`. Why: dev mode hides a production failure and creates inconsistent auth behavior. Recommendation: use `createHeaders()` or a shared upload/stream client. Effort: S. Blast radius: copilot client only.

**S4. WebSocket fan-out is unsafe across strict/non-strict modes**  
Category: Security. Severity: High. Evidence: `server/websocket.ts:639` defaults missing orgs to `SYSTEM_ORG_ID`; strict mode drops system events, non-strict subscribes tenants to system namespace at `:395`. Why: events either leak broadly or disappear in prod. Recommendation: require explicit `orgId` for tenant events and reserve system broadcasts for non-sensitive operational notices. Effort: M. Blast radius: real-time features.

**S5. Upload and prompt-injection defenses are shallow**  
Category: Security. Severity: Medium. Evidence: upload accepts `application/octet-stream` by extension in `agent/interfaces/routes.ts:72`; malware scan disabled by default in `rag/security/types.ts`; custom org prompt is appended raw in `agent/domain/system-prompt.ts:42`. Why: uploaded docs can become prompt/control input. Recommendation: add AV/content scanning, document-instruction isolation, and prompt provenance labels. Effort: M. Blast radius: KB/copilot.

**4. Performance & Cost**
**P1. Telemetry batching still writes row-by-row**  
Category: Performance & cost. Severity: High. Evidence: `server/telemetry-batch-writer.ts:447` slices batches, then `Promise.all` calls `createTelemetryReading` per row; drops/evictions are logged but not operationally surfaced. Why: high DB cost and silent PdM signal loss. Recommendation: use bulk insert/upsert per batch and emit dropped-reading metrics/alerts. Effort: M. Blast radius: telemetry ingest.

**P2. PDM dashboard scans too much data repeatedly**  
Category: Performance & cost. Severity: Medium. Evidence: `get-dashboard.use-case.ts:15` calls `getRiskQueue` three times; `pdm-postgres.repository.ts:224` reads all predictions then filters in JS. Why: dashboard cost grows with historical predictions. Recommendation: SQL aggregate by status/severity with limits and date windows. Effort: S/M. Blast radius: PDM dashboard only.

**P3. New copilot conversations bypass token budget check**  
Category: Performance & cost. Severity: Medium. Evidence: `AgentOrchestrator.initRun` checks budget only when `conversationId` exists at `orchestrator.ts:377`; new conversations are created at `:388` without daily/monthly check. Why: users can start many new conversations and bypass configured limits. Recommendation: check org daily/monthly budget before creating a conversation. Effort: S. Blast radius: agent service.

**5. Reliability**
**R1. Offline conflict resolution is a stub**  
Category: Reliability & operations. Severity: High. Evidence: `server/conflict-resolution-service.ts:20` always returns no conflicts; sync routes call it in `server/domains/sync/routes.ts:173`. Why: offline concurrent edits can be overwritten or falsely accepted. Recommendation: implement version-vector/field conflict detection and persist `sync_conflicts` with org scoping. Effort: L. Blast radius: offline sync protocol.

**R2. Sync/event outbox is not tenant-safe or failure-visible**  
Category: Reliability & operations. Severity: Medium. Evidence: `sync_outbox` has no `org_id` in `shared/schema/sync.ts:55`; `publishEvent` swallows failures at `server/sync-events.ts:166`. Why: lost events are hard to detect and events cannot be safely partitioned by tenant. Recommendation: move to `event_outbox` with `orgId`, retry state, and DLQ metrics. Effort: M. Blast radius: realtime/offline integrations.

**R3. Training runs are in-process and restart-fragile**  
Category: Reliability & operations. Severity: High. Evidence: `training-pipeline.service.ts:61` calls `executeRunAsync` without durable queue; constructor defaults to `StubTrainingRunner`. Why: process restarts can leave runs stuck and model promotion lineage unclear. Recommendation: run training via durable worker/job queue with resumable state. Effort: L. Blast radius: ML platform.

**6. Operational Workflows**
**W1. PDM schedule cannot be adjusted in place**  
Category: End-user operational workflows. Severity: Medium. Evidence: `TaskDetailPanel.tsx:230` disables “Move Task”; `schedule-view/index.tsx:181` says write API is pending. Why: planners must leave the schedule and edit a linked work order manually. Recommendation: add schedule-write endpoint with override/audit reason. Effort: M. Blast radius: PDM scheduling.

**W2. Work-order closeout relies on free text for operational facts**  
Category: End-user operational workflows. Severity: Medium. Evidence: closeout fields in `WorkOrderCloseoutWizard.tsx:117-139`; parts are free text at `:126`; completion maps notes at `useWorkOrdersPageData.ts:213`. Why: inventory, evidence, labor, and model feedback can drift from reality. Recommendation: bind closeout to checklist completion, part usage, attachments, and PdM outcome labels. Effort: M. Blast radius: work orders/inventory/PdM feedback.

**W3. Smart replenishment can create partial PRs**  
Category: End-user operational workflows. Severity: Medium. Evidence: `LowStockReplenishmentPanel.tsx:122` creates PR, then loops item POSTs at `:129`. Why: mid-loop failure leaves incomplete procurement drafts. Recommendation: add atomic batch create endpoint and supplier grouping. Effort: S/M. Blast radius: purchasing API.

**W4. Crew scheduling reports gaps but does not resolve them**  
Category: End-user operational workflows. Severity: Medium. Evidence: unfilled positions are displayed in `enhanced-schedule-results-card.tsx:265`, but no guided fix/apply action follows. Why: planner still has to manually infer substitutions or rule relaxations. Recommendation: add “resolve unfilled” actions: relax constraint, request crew, swap assignment, or accept exception with audit. Effort: M. Blast radius: crew scheduler.

**7. AI/ML Quality**
**M1. Training lifecycle is partly mocked**

Why: model quality, reproducibility, and promotion decisions are not trustworthy. Recommendation: require real runner in production and persist dataset hash, code version, params, metrics, artifact digest. Effort: L. Blast radius: ML training/deploy.

**M2. Inference fallback is not operationally explicit**

Blast radius: PdM predictions.
