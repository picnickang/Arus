Your goal is to DESIGN, VALIDATE, and IMPLEMENT a unified system for:

- Deck Logbook (DLB)
- Engine Room Logbook (ER)
- Compliance & Alerts (operations + regulatory-style checks)
- Email notifications
- UX flow and wireframes for logbooks & compliance

PHASE 1 — UNIFIED LOGBOOK DOMAIN MODEL (DECK + ENGINE)
==========================================================

We want a consistent, unified logbook model that supports BOTH:

- Deck Log Book (DLB)
- Engine Room Log Book (ER)

Logbooks must be:
- Per organization
- Per vessel
- Per date
- Per watch (standard watch blocks e.g. 00–04, 04–08, 08–12, 12–16, 16–20, 20–24)
- Compatible with offline-first (SQLite) and online Postgres modes

Design or refine schema (for both SQLite and PostgreSQL):

logbook_days
id
org_id
vessel_id
date (UTC or vessel local, but consistent)
type ('DECK', 'ENGINE')
status ('draft', 'submitted', 'locked')
created_by
submitted_by
submitted_at
locked_by
locked_at
remarks
compliance_status ('ok', 'warning', 'violation', 'pending_review')

logbook_entries
id
logbook_day_id (FK -> logbook_days)
watch (string or enum: "00-04", "04-08", etc.)
time (timestamp)
created_by
source ('manual', 'auto_telemetry', 'auto_system')
category ('navigation', 'engine', 'alarm', 'maintenance', 'crew', 'bunkering', 'incident', etc.)
payload JSONB (structured fields depending on type; you may define subtypes)
linked_equipment_ids (array or join table)
linked_work_order_ids (array or join table)
linked_alert_ids (array or join table)

Optionally:
logbook_signatures
id
logbook_day_id
role ('Master', 'ChiefEngineer', 'Watchkeeper', etc.)
signed_by_user_id
signed_at
signature_hash / signature_metadata

Requirements:
- Strong vessel + org scoping
- Foreign keys enforced
- Cascade delete only for drafts (never delete locked historical records)
- Able to support multiple log types (deck + engine) with shared framework but separate views

Ensure the schema aligns with the existing database structure and naming conventions.

PHASE 2 — DECK LOGBOOK IMPLEMENTATION / ALIGNMENT
==========================================================

Using the unified model, implement or refactor the Deck Logbook:

1. Map to DECK LOG BOOK UPDATE.xlsx:
- Identify key fields: position, course, speed, weather, remarks, arrivals/departures, incidents, etc.
- Define structured payloads in logbook_entries.payload for:
• Navigation entries
• Port operations
• Weather reports
• Incidents / remarks

2. Auto-population from:
- Telemetry (speed, heading, position if available)
- Mission / voyage module (if in system)
- Work orders closed during the watch
- Alerts and compliance findings

3. UI:
- Deck Logbook page per vessel
- Date selector
- Watch selector
- Table-like form per watch with:
• Time rows
• Relevant fields (position, speed, etc.)
• Auto-fill buttons
- Buttons:
• Save draft
• Submit
• Lock & sign (role: Master)
- Show telemetry mini-graphs or mini-cards for the watch period where available.

4. Export:
- XLSX & PDF exports that mirror DECK LOG BOOK UPDATE.xlsx layout as closely as reasonable.
- Filenames like:
DECK_LOGBOOK_{VESSEL}_{YYYYMMDD}.xlsx

PHASE 3 — ENGINE LOGBOOK (ER) IMPLEMENTATION / ALIGNMENT
==========================================================

Using the same unified model, implement or refactor the Engine Room Logbook (ER):

1. Align with ER LOG BOOK.xlsx:
- ME RPM, load, pressures, temps
- DG generator load, voltage, frequency
- Fuel consumption, running hours
- Alarms, trips, maintenance actions
- Bunkering, tank levels (if defined in template)

2. Telemetry auto-population:
- For each watch and time bucket derive:
• Averages or snapshots from engine/generator telemetry
• Running hours from equipment health
• Any active alarms for the period
- Fill logbook_entries with category='engine' and structured payloads.

3. UI:
- Engine Logbook page per vessel
- Same pattern: date + watch + table
- Auto-populate actions
- Inline display of alarms and work orders for that time window
- Lock & sign controls (role: Chief Engineer)

4. Export:
- XLSX & PDF matching ER LOG BOOK.xlsx format as closely as feasible.
- Filenames like:
ER_LOGBOOK_{VESSEL}_{YYYYMMDD}.xlsx

Ensure deck & engine logbooks stay separate in the UI but share backend infra.

PHASE 4 — COMPLIANCE & ALERT ENGINE (OPS + REGULATORY STYLE)
==========================================================

Create or extend a Compliance & Alerts engine that:

1. Defines rules on top of:
- Logbooks (deck + engine)
- Telemetry data (e.g., over-temp, low pressure, overspeed)
- Maintenance data:
• Overdue work orders
• Repeated deferment
- Crew scheduler & hours of rest:
• Violations based on rest rules
- Sync status:
• Missing logbook days or watches

2. Rule types:
- Operational:
• Missing deck/engine watch entries
• Critical telemetry events not acknowledged
- Safety / compliance:
• Rest hour violations (based on IMO-style guidelines)
• Required log entries missing for port departures/arrivals
• Overdue safety-critical maintenance
- Data integrity:
• Incomplete logbooks marked but locked
• Conflicting entries

3. Implementation:
- Define a rules engine (code-driven rule set is fine)
- Evaluate rules in:
• Background jobs (where enabled)
• On-demand when logbooks are submitted/locked
- Store results as:
compliance_findings
id, org_id, vessel_id, date, source_type ('logbook', 'telemetry', 'crew', 'maintenance'),
severity ('info','warning','critical'),
code (string identifier),
message,
linked_logbook_day_id,
linked_entry_ids,
linked_equipment_ids,
linked_crew_ids,
status ('open','acknowledged','resolved','suppressed')

Note: This is NOT a legal implementation of SOLAS/ISM/MARPOL, but it should be architected to *approximate* operational expectations and be easy to evolve.

PHASE 5 — EMAIL & NOTIFICATION INTEGRATION
==========================================================

Ensure the compliance & alert engine can trigger notifications.

1. Check for existing email / notification infrastructure:
- If exists, integrate with it.
- If not, design a simple adapter for an email service (e.g. SMTP, SendGrid placeholder).

2. Notification Settings UI:
- Under Settings → Notifications / Compliance:
• Allow configuration of:
- Critical logbook issues (e.g. missing watches, un-locked logbooks after X hours)
- Critical engine/telemetry alerts (over-temp, low oil pressure, etc.)
- Crew rest violations
- Overdue safety-critical work orders
• Allow specifying:
- Recipient email(s) per org and optionally per vessel
- Thresholds (e.g. only send severity >= 'critical')

3. Email contents should include:
- Vessel name
- Timestamp
- Alert type, severity, message
- Link or reference to logbook day / equipment / crew involved
- Status (open/acknowledged/resolved)

Make sure:
- Email sending is resilient (queued if offline)
- It respects offline-first architecture: store pending notifications in SQLite and sync later.
