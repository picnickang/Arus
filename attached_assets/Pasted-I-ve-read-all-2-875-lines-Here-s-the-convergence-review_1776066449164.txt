I’ve read all 2,875 lines. Here’s the convergence review.
The convergence infrastructure is strong — genuinely well-built. Three guardrail scripts that run as CI gates, a 331-line integration test suite hitting 20+ endpoint groups, a dual-schema smoke test that validates PG/SQLite parity down to column names, and a domain router registry that declaratively manages 70+ route modules. This is the kind of infrastructure that makes ChatGPT’s “drift” concern manageable rather than terminal.
What’s working well:
The domain-router-registry.ts is the best file in the codebase. 70+ domains registered declaratively, two registration modes clearly documented, dependency injection via getDeps(), and error isolation per domain (a failed import logs an error but doesn’t block other domains). The registerAllDomainRouters function is 25 lines. This is how you scale a modular monolith.
The three guardrail scripts are exactly right for this architecture. check-storage-imports.mjs blocks the old server/storage.ts import path. check-schema-imports.mjs blocks direct @shared/schema/* imports in favor of @shared/schema-runtime (which handles PG/SQLite switching). check-route-registration.mjs blocks rogue app.get("/api/...") calls outside approved files. These are the mechanical enforcement of architectural decisions — not docs that people ignore, but CI gates that fail the build.
The convergence test suite (convergence-validation.sh) covers the full surface: health checks, CRUD for WO/equipment/vessel, PdM domain, tenant isolation, observability, digital twin, KB, agent, and route regressions. The tenant isolation test (missing org header → 401/403) is particularly important for a multi-tenant maritime system.
What I’d flag:
1. The storage facade (createStorageFacade()) is 200+ lines of manual delegation and should be the next retirement target. Every method is a one-liner that forwards to the actual repository — getDevices: (orgId?) => dbDevicesStorage.getDevices(orgId). This was necessary during the migration from the old storage.ts monolith, but now that all the db*Storage modules and domain services are directly importable from repositories.ts, the facade is pure indirection. The guard script blocks imports from the old storage.ts path, but repositories.ts re-exports the facade as export const storage = createStorageFacade() — meaning code that imports storage from repositories.ts is using the facade, while code that imports dbEquipmentStorage directly from repositories.ts is not. Two paths to the same data, again.
Retirement plan: Grep for storage. usage across the server. For each call site, replace storage.getEquipment(orgId, id) with dbEquipmentStorage.getEquipment(orgId, id) (imported from repositories.ts). Once no file imports storage, delete createStorageFacade() and the export const storage line. Add a guard script line that blocks storage. usage just like the current one blocks ../storage imports.
2. The schema guard script allowlist has 120+ entries. Each is a file that directly imports @shared/schema instead of @shared/schema-runtime. The allowlist is honest (it doesn’t hide the debt), but 120 files is a lot. The check-schema-imports.mjs script will catch new violations, but the existing 120 won’t shrink unless someone works through them. The drift burndown monotonic guard (must never exceed 116) is the right mechanism — it means every new file MUST use schema-runtime, and occasionally an old file gets migrated.
3. The inline-routes.ts has two production routes mixed with dev-only endpoints. The MQTT health endpoint and the equipment load-distribution endpoint are production routes that happen to live in the “inline” file instead of in domain modules. The load-distribution endpoint in particular does raw SQL queries against equipment and telemetry — it should be in the vessel-performance or equipment-intelligence domain. The comment says “Production-facing routes have been migrated to the domain router registry. Only dev-only endpoints remain here.” — but that’s not accurate. Two of the four endpoints are production-facing.
4. The shared/schema.ts barrel file is 688 lines of re-exports and standalone Zod schemas. The re-exports are necessary (they’re the public API), but the standalone Zod validation schemas (admin password, telemetry queries, HoR, PdM, etc.) are mixed in with table re-exports. These should live in a shared/validation/ directory, grouped by domain. The barrel should only re-export tables and their insert schemas. This isn’t urgent — it’s a readability issue, not a correctness issue — but it’ll make the schema.ts file harder to navigate as more domains add validation schemas.
5. The repositories.ts has duplicate imports. Lines 1-24 export the db* modules, then lines 35-98 import many of the same modules again for the analyticsDeps, schedulingDeps, and createStorageFacade function. The duplicate imports won’t cause bugs (they’re the same module), but they make the file look messier than it is. When the storage facade is retired, these duplicate imports go away naturally.
Overall assessment: The convergence infrastructure is production-grade. The guardrail scripts, the domain router registry, and the integration test suite are exactly the tools needed to manage a 70+ domain system. The remaining debt (120-file schema allowlist, storage facade, 2 inline production routes) is tracked, bounded, and shrinking. The architecture supports cleanup — you can retire one module at a time without cross-cutting changes. This is a system in controlled convergence, not uncontrolled drift.​​​​​​​​​​​​​​​​
