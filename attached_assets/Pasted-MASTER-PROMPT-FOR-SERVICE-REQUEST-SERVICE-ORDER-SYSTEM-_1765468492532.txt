1️⃣ CONCEPTUAL WORKFLOW



From the user’s perspective:

They create a Service Request (SR) for specific equipment.

From that Service Request, they generate Service Orders (SO) to external service providers.

From the same SR (or underlying Work Order), they can generate Purchase Requests / Purchase Orders for required parts.

There are separate tabs:

Service Requests tab (existing Work Order UI with SR flavor)

Service Orders tab (list + detail)

Purchase Orders tab (list + detail)



Under the hood:

A Service Request is implemented as a Work Order of type “service_request” (and possibly “certificate_renewal” for renewals).

Service Orders are separate entities linked back to that Work Order/Service Request.

Purchase Orders are already their own entities; we just ensure:

They’re linkable to the Work Order / Service Request.

They have their own tab.

3️⃣ SERVICE ORDER DATA MODEL & BEHAVIOR



Create or extend a Service Order table/model (name to match existing conventions):



Fields:

service_orders:
  id
  so_number
  work_order_id       // FK to Work Order (Service Request)
  service_provider_id // FK to supplier/service provider
  status: "planned" | "sent" | "in_progress" | "completed" | "cancelled"

  scheduled_start_at   // planned service start datetime
  scheduled_end_at     // planned service end datetime
  actual_start_at      // when service actually started
  actual_end_at        // when service actually ended

  expected_duration_hours?   // optional hint
  service_scope: text        // high-level scope
  service_details: text      // what was actually done

  cancelled_at?
  cancelled_by?
  cancellation_reason?

  created_at
  updated_at
Status transitions

planned → sent

sent → in_progress

in_progress → completed

planned/sent/in_progress → cancelled

completed and cancelled are locked (no edits to core fields except admin tooling if it exists).



Service time

Service time is derived from actual_start_at / actual_end_at.

UI must support:

Setting scheduled start/end.

Setting actual start/end.

Displaying duration in hours.

Showing progress/overdue status visually.
