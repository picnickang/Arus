Replit Prompt: Performance Test ARUS Ingestion Layer (v3)

You are a performance engineer. Build a load/performance test suite for the ARUS v3 ingestion pipeline:

C# Agent → SQLite(raw_frames) → Node Bridge → Decoder → TelemetryBatchWriter.writeBatch(source=sqlite-bridge) → Postgres

Goal: quantify capacity and find bottlenecks (SQLite write, bridge decode, Postgres bulk insert, backpressure).

Non-negotiable rules
	1.	Tests must be repeatable and configurable via env vars.
	2.	Collect metrics at each stage: agent write rate, SQLite backlog, bridge decode rate, Postgres commit rate, end-to-end lag.
	3.	Must support failure mode tests: Postgres down, slow Postgres, bridge restart, agent restart.
	4.	Must produce a single summary report (JSON + human-readable) with key KPIs and recommended tuning knobs.
	5.	Do not change production logic except adding safe instrumentation/metrics hooks.

Step 1 — Add a deterministic load generator (Agent side)

In the C# agent, implement a dev-only “SimulatedSource” (implements IIngestSource) that generates frames at a target rate:

Env vars:
	•	ARUS_SIM_ENABLE=true|false
	•	ARUS_SIM_PROTOCOL=J1939
	•	ARUS_SIM_FPS=1000 (frames/sec)
	•	ARUS_SIM_SOURCES=1 (e.g., can0..canN)
	•	ARUS_SIM_DURATION_S=60

Behavior:
	•	Generate valid J1939 payload envelope v1:
[canId:u32le][dlc:u8][data]
	•	Use a fixed set of PGNs that your decoder supports (include at least one that produces a TelemetryReading).
	•	Track: framesGenerated, framesEnqueued, framesWritten, queueDepth, lastFrameTime.

Expose these in:
	•	heartbeat JSON
	•	SubscribeStats gRPC stream

Step 3 — Create a dedicated load test runner

Add a Node script:

server/tools/ingestion-loadtest.ts

It should:
	1.	Start/verify the agent simulator (via gRPC StartSource or env-enabled auto-start).
	2.	Ensure SQLite schema exists.
	3.	Start bridge (or ensure it’s running).
	4.	Run for ARUS_LOADTEST_DURATION_S seconds.
	5.	Sample stats every 1s:
	•	agent stats (gRPC or heartbeat)
	•	SQLite counts: total frames, frames last 10s, max(id)
	•	cursor last_id
	•	Postgres commits + inserted rows (or writer metrics)
	6.	Output:
	•	reports/ingestion-loadtest-YYYYMMDD-HHMMSS.json
	•	human-readable summary to console

Env vars for runner:
	•	ARUS_SQLITE_PATH (required)
	•	ARUS_LOADTEST_DURATION_S=60
	•	ARUS_LOADTEST_TARGET_FPS=1000
	•	ARUS_LOADTEST_ACCEPTABLE_BACKLOG=5000
	•	ARUS_LOADTEST_MAX_LAG_MS=60000
	•	ARUS_LOADTEST_POSTGRES_MODE=normal|slow|down
	•	slow: inject artificial delay per commit or throttle pool
	•	down: simulate connection refusal

Step 4 — Performance scenarios (must implement)

Implement these scenarios and report separately:

Scenario A: Baseline throughput
	•	Postgres normal
	•	FPS sweep: 250, 500, 1000, 2000, 5000 (stop when backlog diverges)
Report:
	•	sustainable FPS (steady backlog)
	•	avg/95p commit latency
	•	CPU/mem peaks (best effort)

Scenario B: Postgres slow
	•	Add commit delay (e.g., 100ms/250ms) or throttle pool to 1 connection
Report:
	•	backlog growth rate
	•	recovery time after removing slowness

Scenario C: Postgres down
	•	Force write failures for 30s
Report:
	•	cursor does not advance
	•	backlog grows
	•	after restore: catch-up rate and time to recover

Scenario D: Bridge restart mid-stream
	•	Kill and restart bridge at T+30s
Report:
	•	no data loss
	•	cursor resumes correctly
	•	no double writes (idempotency stable)

Scenario E: Agent restart mid-stream
	•	Restart agent at T+30s
Report:
	•	raw_frames continuity
	•	bridge handles pause/resume

Step 5 — Postgres writeBatch performance requirements

Verify TelemetryBatchWriter.writeBatch() is bulk inserting efficiently:
	•	must not do per-row await query() loops
	•	must do batched VALUES or COPY approach
	•	must use ON CONFLICT (idempotency_key) DO NOTHING

Add a test/assertion in perf runner:
	•	if writeBatch uses per-row queries, fail the run with a warning.
