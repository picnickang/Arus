# ARUS Source-Aware UI Alignment Prompt

Use the uploaded source code and attached UI preview images as the source of truth.

## Source review summary
The current ARUS frontend already contains the correct foundation for the new UI direction:

- `client/src/pages/portal-login.tsx` already implements the Admin/User portal split.
- `client/src/pages/home.tsx` already renders portal-aware branches.
- `client/src/pages/feedback.tsx` already contains the user Feedback / Flags flow with local/session storage.
- `client/src/components/BottomNav.tsx` is already policy-driven.
- `client/src/application/navigation/role-navigation-policy.ts` is already the correct single source of truth for role-to-portal and role-to-navigation mapping.
- `client/src/domain/navigation/types.ts` already defines the clean domain types.
- `client/src/features/workflow/components/WorkflowCommandCenter.tsx` and `RoleTodayPanel.tsx` already provide the start of the command-center UX.
- `client/src/index.css` already contains dark theme tokens, but the app still needs a more intentional ARUS command-center visual layer matching the preview images.

Therefore: do NOT rewrite the frontend from scratch. Align and extend the existing architecture.

# Mission

Align the existing ARUS frontend with the attached dark-mode operational-intelligence UI previews.

The redesign should feel like:

- a maritime operations command center
- an attention-management system
- an operational intelligence platform

It should NOT feel like:

- a raw telemetry console
- a cluttered enterprise module grid
- a generic analytics dashboard

---

# Non-negotiable constraints

Preserve:

- existing backend APIs
- existing route behavior where possible
- strict TypeScript cleanliness
- `explicit-any = 0`
- documented cast baseline
- RBAC behavior
- tenant isolation
- hexagonal frontend architecture
- existing tests and launch-readiness hardening

Do not introduce:

- `any`
- `as any`
- `@ts-ignore`
- `eslint-disable`
- fake persistence
- fake backend success
- broken routes
- removed functionality
- large unreviewed rewrites

# Required architecture

Keep the hexagonal frontend boundary:

## domain
Use for pure types only:

- portal types
- navigation types
- command-center card models
- attention item models
- reporting workflow models

Current relevant file:

- `client/src/domain/navigation/types.ts`

## application
Use for policy and orchestration:

- role-to-navigation policy
- role-to-portal mapping
- command-center composition
- user-dashboard composition
- feedback validation/submission model
- AI recommendation card shaping

Current relevant files:

- `client/src/application/navigation/role-navigation-policy.ts`
- `client/src/application/feedback/feedback-submission.ts`

Add new source-aware modules here if needed, for example:

- `client/src/application/command-center/command-center-view-model.ts`
- `client/src/application/command-center/admin-command-center-policy.ts`
- `client/src/application/user-dashboard/user-dashboard-view-model.ts`

## infrastructure
Use for API/session/storage adapters only:

- API clients
- session/localStorage adapters
- websocket adapters
- feature flags

Do not put policy in React components.

## interfaces
Use for React rendering only:

- pages
- layouts
- cards
- responsive views

Current relevant files:

- `client/src/pages/portal-login.tsx`
- `client/src/pages/home.tsx`
- `client/src/pages/feedback.tsx`
- `client/src/components/BottomNav.tsx`
- `client/src/components/navigation/SwitchPortalButton.tsx`
- `client/src/features/workflow/components/WorkflowCommandCenter.tsx`
- `client/src/features/workflow/components/RoleTodayPanel.tsx`

React components must render view models from the application layer; they must not decide RBAC or navigation policy themselves.
