You are a senior product/finance systems analyst + full-stack engineer. You are working inside an existing Node.js/TypeScript Predictive Maintenance (PdM) app (ARUS). Your goal is to determine whether the application currently contains enough information (in database schema, API endpoints, and codebase) to support the “monetary aspect” of PdM KPIs and screens, and if not, specify exactly what else is required.

====================================================
A) DEFINE “MONETARY ASPECT” (WHAT WE WANT TO SHOW)
====================================================
The PdM UI/analytics requires these money-related outputs:

1) Avoided downtime (hours + $) over last 30 days
2) Maintenance spend forecast ($) for next 30 days
3) Expected downtime forecast (hours) for next 30 days
4) ROI / savings summary by vessel, asset class, and failure mode (optional but ideal)
5) Cost breakdown: labor, parts, contractors, penalties / opportunity cost (optional)

These outputs must be computable from data in the repo or from clearly defined external inputs/config.

====================================================
B) REQUIRED INPUT DATA (WHAT MUST EXIST SOMEWHERE)
====================================================
Check whether the repo has each of the following categories, and where:

1) Downtime cost model inputs (minimum viable):
- downtime_cost_rate: $/hour (by vessel OR vessel class OR contract)
- downtime_impact_multiplier (full stop vs degraded)
- average_downtime_hours by asset class and/or failure mode
- penalty rules or lost revenue rules (if any)

2) Maintenance cost inputs:
- work order costs: labor_hours, labor_rate, parts_cost, contractor_cost
- OR average repair cost by failure mode/asset class (fallback)
- maintenance records linking: asset_id/equipment_id, vessel_id, timestamps, failure_mode (if available)

3) Risk/probability inputs:
- p_fail_7d / p_fail_14d / p_fail_30d OR survival curve outputs OR a calibrated mapping from risk score to probability
- RUL estimates (ideally quantiles)

4) Mitigation attribution data:
- alert_id, created_at, resolved_at
- work_order_id, created_at, started_at, completed_at
- link table between alerts and work orders OR a direct foreign key
- logic defining “mitigated before expected failure window”

5) Baseline & historical distributions (ideal, not mandatory):
- MTBF/MTTF by asset class
- historical downtime durations by failure mode
- historical repair cost distributions

6) Currency and accounting:
- currency_code (per org)
- multi-currency support (optional)
- period boundaries used in reporting

====================================================
C) STEP 1 — DISCOVER CURRENT REPO DATA
====================================================
Search the repo for:
- DB tables and columns: work_orders, maintenance_records, parts, inventory, invoices, costs, rates, downtime, penalties, charter, day_rate
- PdM tables: pdm_alerts, pdm_alert_evidence, pdm_predictions
- Any config files: “cost”, “rate”, “forecast”, “roi”, “downtime”
- API endpoints: finance, maintenance costs, work order totals, spend summaries
- Any UI components showing cost, ROI, downtime, forecast

Output:
1) A list of relevant files (paths) and what they contain
2) A DB schema summary of the money-related tables/columns found
3) A map: “Monetary output -> where the required inputs come from (table/endpoint/config)”

====================================================
D) STEP 2 — GAP ANALYSIS
====================================================
Create a gap table:

For each required input category in section B:
- Present? (Yes/Partial/No)
- Evidence (table/column/file path)
- Quality issues (missing keys, no failure_mode link, no timestamps, etc.)
- What is missing exactly (field names and suggested data types)
- Suggested best source (DB table vs config vs UI admin settings)

IMPORTANT:

- If work orders lack cost fields, propose minimal cost fields or a cost breakdown table.
- If risk outputs are not probabilistic, propose mapping risk score -> probability with calibration stub.

====================================================
E) STEP 3 — RECOMMENDED MINIMUM ADDITIONS (IF MISSING)
====================================================
If inputs are missing, propose the minimal additions needed to compute outputs:

1) Tables (idempotent SQL preferred):
- org_cost_config (currency, downtime_cost_rate, labor_rate_default, etc.)
- failure_mode_cost_defaults (avg_repair_cost, avg_downtime_hours)
- work_order_costs (or columns on work_orders)

2) Endpoints:
- GET /api/pdm/finance-summary?from&to&filters...
