### Module Breakdown

|Module         |Responsibility                                                             |Key Entities                                                 |
|---------------|---------------------------------------------------------------------------|-------------------------------------------------------------|
|**maintenance**|Assets, work orders, telemetry, failure history, predictive alerts         |Asset, WorkOrder, MaintenanceEvent, SensorReading, Prediction|
|**inventory**  |Parts, tools, consumables, locations, reservations, purchasing             |InventoryItem, InventoryLocation, Reservation, PurchaseOrder |
|**scheduling** |Technician/crew management, shifts, assignments, skill matrix, availability|Technician, Crew, Shift, Assignment, Skill                   |
|**agent**      |AI orchestration, signal processing, diagnosis, autonomous action execution|AgentRun, AgentAction, ApprovalRequest, Signal               |
|**shared**     |Event bus, audit trail, auth, common types, utilities                      |AuditLog, User, EventDefinitions                             |
|**db**         |Drizzle schema, migrations, connection management, seed data               |Schema files, migration runner                               |

### Agent Position: Orchestrator (Not Embedded)

The agent is a **standalone orchestrator** that calls into the other modules via their service interfaces — not embedded within any single module. This keeps the agent testable, its permissions auditable, and prevents coupling domain logic to LLM behavior.

### Recommended Libraries

|Concern                 |Library                           |Why                                                                  |
|------------------------|----------------------------------|---------------------------------------------------------------------|
|ORM / Query Builder     |**Drizzle ORM** (already in use)  |First-class SQLite + PostgreSQL support, type-safe, migration tooling|
|Validation              |**zod**                           |Schema-first, pairs with Drizzle and Express                         |
|Event Bus               |**eventemitter3** or Node built-in|Typed wrapper, no external dependency needed                         |
|Scheduling (cron)       |**node-cron** or **croner**       |Lightweight background jobs                                          |
|LLM                     |**openai** SDK                    |Function-calling for agent tools                                     |
|HTTP client (ML sidecar)|**undici** (Node built-in)        |No extra dependency                                                  |
|ID generation           |**nanoid** or **cuid2**           |URL-safe, no collision, works everywhere                             |
|Date handling           |**date-fns**                      |Tree-shakeable, no mutation                                          |

## Step 2: Data Models and Database Schema

### Cross-Database Strategies

|Concern           |Strategy                                              |Notes                                                                                                                             |
|------------------|------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|
|**IDs**           |`TEXT` primary keys with `nanoid()` or `cuid2()`      |Avoid autoincrement — not portable, bad for distributed/offline sync. Generate in app layer.                                      |
|**Timestamps**    |`TEXT` in ISO-8601 format (`YYYY-MM-DDTHH:mm:ss.sssZ`)|SQLite has no native timestamp. Use `new Date().toISOString()` in app. PostgreSQL can cast TEXT to TIMESTAMP if you migrate later.|
|**Enums**         |`TEXT` column + Drizzle `.$type<>()` + zod validation |Don’t use PostgreSQL `CREATE TYPE` or SQLite `CHECK`. Enforce in app.                                                             |
|**JSON fields**   |`TEXT` column, store `JSON.stringify()`, parse in app |Both DBs support this. PostgreSQL `jsonb` is an optional upgrade later.                                                           |
|**Booleans**      |`INTEGER` (0/1) for SQLite, `BOOLEAN` for PostgreSQL  |Drizzle handles this per-dialect in schema files.                                                                                 |
|**Decimals/money**|`INTEGER` storing cents, or `REAL`/`NUMERIC`          |Avoid floating point for money. Store cents as integers.                                                                          |

### Drizzle Schema (SQLite-first, PostgreSQL annotations)

```typescript
// db/schema/common.ts
import { sqliteTable, text, integer, real } from 'drizzle-orm/sqlite-core';
import { nanoid } from 'nanoid';

// Helper: generate a prefixed ID
export const genId = (prefix: string) => `${prefix}_${nanoid(16)}`;

// Reusable audit columns pattern
export const auditColumns = {
  createdAt: text('created_at').notNull().$defaultFn(() => new Date().toISOString()),
  updatedAt: text('updated_at').notNull().$defaultFn(() => new Date().toISOString()),
  createdBy: text('created_by'),
  updatedBy: text('updated_by'),
};
```

### Migration Strategy

Use `drizzle-kit` with separate config files per dialect:

```typescript
// drizzle.config.ts
import { defineConfig } from 'drizzle-kit';

const isSqlite = process.env.DB_DIALECT !== 'postgresql';

export default defineConfig({
  schema: './src/db/schema/*.ts',
  out: isSqlite ? './drizzle/sqlite' : './drizzle/pg',
  dialect: isSqlite ? 'sqlite' : 'postgresql',
  dbCredentials: isSqlite
    ? { url: process.env.DATABASE_URL || './data/arus.db' }
    : { url: process.env.DATABASE_URL! },
});
```

Maintain **two migration directories** (`drizzle/sqlite/`, `drizzle/pg/`). Generate migrations per dialect. This avoids fighting SQL syntax differences in a single migration set.

```typescript
  async generateWorkOrderNumber(): Promise<string> {
    const year = new Date().getFullYear();
    const prefix = `WO-${year}-`;

    // Count existing WOs this year — works in both SQLite and PG
    const result = await this.db
      .select({ count: sql<number>`count(*)` })
      .from(workOrders)
      .where(sql`${workOrders.number} LIKE ${prefix + '%'}`);

    const seq = (result[0]?.count ?? 0) + 1;
    return `${prefix}${String(seq).padStart(4, '0')}`;
  }

  async createWorkOrder(input: CreateWorkOrderInput & { number: string }) {
    const [wo] = await this.db.insert(workOrders).values({
      ...input,
      requiredSkills: input.requiredSkills ? JSON.stringify(input.requiredSkills) : null,
    }).returning();
    return wo;

    return this.db.query.workOrders.findFirst({
      where: eq(workOrders.id, id),
    });

    const [updated] = await this.db
      .update(workOrders)
      .set({ status, updatedAt: new Date().toISOString(), updatedBy })
      .where(eq(workOrders.id, id))
      .returning();
    return updated;
```

```typescript
// modules/inventory/inventory.service.ts
import { eq, and, sql } from 'drizzle-orm';
import { AppDb } from '../../db/client';
import { inventoryStock, inventoryReservations } from '../../db/schema/inventory';
import { eventBus } from '../../shared/event-bus';

export class InventoryService {
  constructor(private db: AppDb) {}

  async reserveForWorkOrder(params: {
    itemId: string;
    locationId: string;
    workOrderId: string;
    quantity: number;
    userId?: string;
  }) {
    // Find stock record
    const [stock] = await this.db
      .select()
      .from(inventoryStock)
      .where(
        and(
          eq(inventoryStock.itemId, params.itemId),
          eq(inventoryStock.locationId, params.locationId),
        )
      );

    if (!stock) throw new Error('No stock record for this item at this location');
    if (stock.quantityAvailable < params.quantity) {
      throw new Error(
        `Insufficient stock: ${stock.quantityAvailable} available, ${params.quantity} requested`
      );
    }

    // In a transaction: update stock and create reservation
    // SQLite and PG both support this through Drizzle's transaction API
    const result = await this.db.transaction(async (tx) => {
      // Decrement available, increment reserved
      await tx
        .update(inventoryStock)
        .set({
          quantityReserved: sql`${inventoryStock.quantityReserved} + ${params.quantity}`,
          quantityAvailable: sql`${inventoryStock.quantityAvailable} - ${params.quantity}`,
          updatedAt: new Date().toISOString(),
        })
        .where(eq(inventoryStock.id, stock.id));

      // Create reservation record
      const [reservation] = await tx
        .insert(inventoryReservations)
        .values({
          stockId: stock.id,
          workOrderId: params.workOrderId,
          quantity: params.quantity,
          status: 'reserved',
        })
        .returning();

      return reservation;
    });

    eventBus.emit('inventory.reserved', {
      reservationId: result.id,
      workOrderId: params.workOrderId,
      itemId: params.itemId,
      quantity: params.quantity,
    });

    // Check for low stock
    const [updatedStock] = await this.db
      .select()
      .from(inventoryStock)
      .where(eq(inventoryStock.id, stock.id));

    // Would need to join to inventoryItems to get reorderPoint;
    // simplified here for clarity
    if (updatedStock.quantityAvailable <= 5) {
      eventBus.emit('inventory.low_stock', {
        itemId: params.itemId,
        locationId: params.locationId,
        available: updatedStock.quantityAvailable,
        reorderPoint: 5,
      });
    }

    return result;
  }
}
```

```typescript
// modules/scheduling/scheduling.service.ts
import { eq, and, gte, lte, sql, not, inArray } from 'drizzle-orm';
import { AppDb } from '../../db/client';
import { technicians, technicianSkills, shifts, assignments } from '../../db/schema/scheduling';
import { eventBus } from '../../shared/event-bus';

export class SchedulingService {
  constructor(private db: AppDb) {}

  async assignTechnician(params: {
    workOrderId: string;
    technicianId: string;
    scheduledStart: string;
    scheduledEnd: string;
    userId?: string;
  }) {
    // Verify technician exists and is active
    const tech = await this.db.query.technicians.findFirst({
      where: eq(technicians.id, params.technicianId),
    });
    if (!tech || tech.status !== 'active') {
      throw new Error('Technician not found or not active');
    }

    // Check for conflicts: overlapping assignments
    const conflicts = await this.db
      .select()
      .from(assignments)
      .where(
        and(
          eq(assignments.technicianId, params.technicianId),
          not(inArray(assignments.status, ['completed', 'declined'])),
          // Overlap check: existing.start < new.end AND existing.end > new.start
          sql`${assignments.scheduledStart} < ${params.scheduledEnd}`,
          sql`${assignments.scheduledEnd} > ${params.scheduledStart}`,
        )
      );

    if (conflicts.length > 0) {
      throw new Error(`Technician has ${conflicts.length} conflicting assignment(s)`);
    }

    const [assignment] = await this.db
      .insert(assignments)
      .values({
        workOrderId: params.workOrderId,
        technicianId: params.technicianId,
        scheduledStart: params.scheduledStart,
        scheduledEnd: params.scheduledEnd,
        status: 'assigned',
      })
      .returning();

    eventBus.emit('assignment.created', {
      assignmentId: assignment.id,
      workOrderId: params.workOrderId,
      technicianId: params.technicianId,
    });

    return assignment;
  }

  async findAvailableTechnicians(params: {
    start: string;
    end: string;
    requiredSkills?: string[];
  }) {
    // Get all active technicians
    let techList = await this.db
      .select()
      .from(technicians)
      .where(eq(technicians.status, 'active'));

    // Filter by skills if specified
    if (params.requiredSkills && params.requiredSkills.length > 0) {
      const skilled = await this.db
        .select({ technicianId: technicianSkills.technicianId })
        .from(technicianSkills)
        .where(inArray(technicianSkills.skillId, params.requiredSkills));

      const skilledIds = new Set(skilled.map((s) => s.technicianId));
      techList = techList.filter((t) => skilledIds.has(t.id));
    }

    // Filter out those with conflicting assignments
    const busyTechs = await this.db
      .select({ technicianId: assignments.technicianId })
      .from(assignments)
      .where(
        and(
          not(inArray(assignments.status, ['completed', 'declined'])),
          sql`${assignments.scheduledStart} < ${params.end}`,
          sql`${assignments.scheduledEnd} > ${params.start}`,
        )
      );

    const busyIds = new Set(busyTechs.map((b) => b.technicianId));
    return techList.filter((t) => !busyIds.has(t.id));
  }
}
```

```typescript
};

interface AgentSignal {
  type: 'prediction' | 'anomaly' | 'manual';
  sourceId: string;
  assetId: string;
  summary: string;
  severity: 'low' | 'medium' | 'high' | 'critical';
  data?: Record<string, any>;
}

export class AgentOrchestrator {
  private openai: OpenAI;

  constructor(
    private db: AppDb,
    private toolExecutor: AgentToolExecutor,
  ) {
    this.openai = new OpenAI();
  }

  async processSignal(signal: AgentSignal): Promise<string> {
    // 1. Create agent run record
    const [run] = await this.db.insert(agentRuns).values({
      triggerId: signal.sourceId,
      triggerType: signal.type,
      status: 'running',
    }).returning();

    try {
      // 2. Build system prompt
      const systemPrompt = `You are the ARUS maintenance operations agent for a maritime vessel.
You have been triggered by a ${signal.type} signal.

Signal details:
- Asset: ${signal.assetId}
- Severity: ${signal.severity}
- Summary: ${signal.summary}
${signal.data ? `- Data: ${JSON.stringify(signal.data)}` : ''}

Your job:
1. Gather context about the asset and its history using the query tools.
2. Diagnose the likely root cause.
3. Determine what actions are needed (work order, parts, crew assignment).
4. Execute those actions using the available tools.
5. If your confidence is below 70%, or the situation is critical, escalate to a human.

Be methodical. Query first, then act. Explain your reasoning.`;

      // 3. Run the agentic loop
      const messages: OpenAI.ChatCompletionMessageParam[] = [
        { role: 'system', content: systemPrompt },
        { role: 'user', content: `Process this signal and take appropriate action.` },
      ];

      let iterationCount = 0;
      const maxIterations = 10;
      let actionSequence = 0;

      while (iterationCount < maxIterations) {
        iterationCount++;

        const response = await this.openai.chat.completions.create({
          model: 'gpt-4o',
          messages,
          tools: agentToolDefinitions,
          tool_choice: iterationCount === 1 ? 'required' : 'auto',
        });

        const message = response.choices[0].message;
        messages.push(message);

        // If no tool calls, the agent is done reasoning
        if (!message.tool_calls || message.tool_calls.length === 0) {
          // Save final reasoning
          await this.db
            .update(agentRuns)
            .set({
              status: 'completed',
              reasoning: message.content,
              completedAt: new Date().toISOString(),
            })
            .where(eq(agentRuns.id, run.id));
          break;
        }

        // Process each tool call
        for (const toolCall of message.tool_calls) {
          const fnName = toolCall.function.name;
          const fnArgs = JSON.parse(toolCall.function.arguments);

          // Check if approval is needed
          const needsApproval = APPROVAL_RULES[fnName]?.(fnArgs) ?? false;

          // Record the planned action
          const [action] = await this.db.insert(agentActions).values({
            runId: run.id,
            actionType: fnName,
            payload: JSON.stringify(fnArgs),
            status: needsApproval ? 'pending_approval' : 'executing',
            requiresApproval: needsApproval ? 1 : 0,
            sequenceOrder: actionSequence++,
          }).returning();

          let result: any;

          if (needsApproval) {
            // Pause execution, notify humans
            await this.db
              .update(agentRuns)
              .set({ status: 'awaiting_approval' })
              .where(eq(agentRuns.id, run.id));

            result = {
              status: 'awaiting_approval',
              message: `Action ${fnName} requires human approval before execution.`,
              actionId: action.id,
            };
          } else {
            // Execute immediately
            try {
              result = await this.toolExecutor.execute(fnName, fnArgs, run.id);
              await this.db
                .update(agentActions)
                .set({
                  status: 'completed',
                  result: JSON.stringify(result),
                  executedAt: new Date().toISOString(),
                })
                .where(eq(agentActions.id, action.id));
            } catch (err: any) {
              result = { error: err.message };
              await this.db
                .update(agentActions)
                .set({
                  status: 'failed',
                  error: err.message,
                })
                .where(eq(agentActions.id, action.id));
            }
          }

          // Feed result back to the LLM
          messages.push({
            role: 'tool',
            tool_call_id: toolCall.id,
            content: JSON.stringify(result),
          });
        }
      }

      return run.id;
    } catch (err: any) {
      await this.db
        .update(agentRuns)
        .set({
          status: 'failed',
          error: err.message,
          completedAt: new Date().toISOString(),
        })
        .where(eq(agentRuns.id, run.id));
      throw err;
    }
  }

  // Called when a human approves a pending action
  async approveAction(actionId: string, approvedBy: string) {
    const [action] = await this.db
      .select()
      .from(agentActions)
      .where(eq(agentActions.id, actionId));

    if (!action || action.status !== 'pending_approval') {
      throw new Error('Action not found or not pending approval');
    }

    const args = JSON.parse(action.payload);
    const result = await this.toolExecutor.execute(action.actionType, args, action.runId);

    await this.db
      .update(agentActions)
      .set({
        status: 'completed',
        result: JSON.stringify(result),
        approvedBy,
        approvedAt: new Date().toISOString(),
        executedAt: new Date().toISOString(),
      })
      .where(eq(agentActions.id, actionId));

    // Check if all actions in this run are now resolved
    const pendingActions = await this.db
      .select()
      .from(agentActions)
      .where(eq(agentActions.runId, action.runId));

    const stillPending = pendingActions.some(
      (a) => a.status === 'pending_approval' || a.status === 'planned'
    );

    if (!stillPending) {
      await this.db
        .update(agentRuns)
        .set({ status: 'completed', completedAt: new Date().toISOString() })
        .where(eq(agentRuns.id, action.runId));
    }

    return result;
  }
}
```

## Step 5: Predictive Maintenance Integration

### Option Comparison

|Option                           |Pros                                                               |Cons                                                   |Difficulty|Best For                   |
|---------------------------------|-------------------------------------------------------------------|-------------------------------------------------------|----------|---------------------------|
|**Python microservice (FastAPI)**|Full ML ecosystem, GPU support, independent scaling, clear boundary|Extra deployment artifact, network calls, more infra   |Medium    |Production with real models|

### Recommendation: Hybrid — Python batch (vessel) + Python sidecar (cloud)

On-vessel (air-gapped Tauri): Run Python batch jobs via `node:child_process` on a cron schedule. Results written directly to the `predictions` table.

Cloud deployment: Run FastAPI sidecar, called from Node via HTTP.

```typescript
// jobs/prediction-runner.ts
import { getDb } from '../db/client';
import { predictions } from '../db/schema/maintenance';
import { eventBus } from '../shared/event-bus';

// For vessel mode: call Python via child_process
import { execFile } from 'node:child_process';
import { promisify } from 'node:util';
const execFileAsync = promisify(execFile);

async function runPredictionBatch(assetIds: string[]) {
  const db = getDb();

  // Option A: Python batch
  const { stdout } = await execFileAsync('python', [
    'ml/predict.py',
    '--assets', assetIds.join(','),
    '--output', 'json',
  ]);

  const results: PredictionResponse[] = JSON.parse(stdout);

  // Store predictions
  for (const result of results) {
    for (const pred of result.predictions) {
      const threshold = getThreshold(pred.type);
      const exceeded = pred.value > threshold;

      const [saved] = await db.insert(predictions).values({
        assetId: result.assetId,
        modelId: result.modelId,
        modelVersion: result.modelVersion,
        predictionType: pred.type,
        value: pred.value,
        confidence: pred.confidence,
        horizon: pred.horizon ?? null,
        explanation: pred.explanation,
        features: JSON.stringify(pred.features),
        threshold,
        exceededThreshold: exceeded ? 1 : 0,
        generatedAt: result.generatedAt,
      }).returning();

      if (exceeded) {
        eventBus.emit('prediction.threshold_exceeded', {
          predictionId: saved.id,
          assetId: result.assetId,
          value: pred.value,
          threshold,
        });
      }
    }
  }
}

function getThreshold(type: string): number {
  // Configurable thresholds
  const defaults: Record<string, number> = {
    failure_probability: 0.7,
    remaining_useful_life: 168,  // hours (7 days)
    anomaly_score: 0.85,
  };
  return defaults[type] ?? 0.8;
}
```

## Step 6: Scheduling Logic

### Algorithm Choice: Greedy Score-Based Assignment

For MVP, a greedy algorithm that scores technicians on multiple weighted criteria is far more practical than constraint solvers or ILP. It’s easy to understand, debug, and evolve.

**Scoring criteria:**

- Skill match (40%): Does the tech have the required skills? Higher proficiency = higher score.
- Availability (30%): Is the tech free during the window? Overlap = disqualified.
- Current workload (20%): Fewer active assignments = higher score.
- Location proximity (10%): Same default location as asset = bonus.

```typescript
// modules/scheduling/scheduling.engine.ts
import { AppDb } from '../../db/client';
import { technicians, technicianSkills, assignments, shifts } from '../../db/schema/scheduling';
import { eq, and, not, inArray, sql, count } from 'drizzle-orm';

interface SchedulingRequest {
  workOrderId: string;
  assetId: string;
  assetLocationId?: string;
  requiredSkills: string[];
  scheduledStart: string;
  scheduledEnd: string;
  priority: string;
}

interface ScoredTechnician {
  technicianId: string;
  firstName: string;
  lastName: string;
  score: number;
  breakdown: {
    skillScore: number;
    availabilityScore: number;
    workloadScore: number;
    proximityScore: number;
  };
}

export class SchedulingEngine {
  constructor(private db: AppDb) {}

  async rankCandidates(request: SchedulingRequest): Promise<ScoredTechnician[]> {
    // 1. Get all active technicians
    const allTechs = await this.db
      .select()
      .from(technicians)
      .where(eq(technicians.status, 'active'));

    // 2. Get their skills
    const allSkills = await this.db.select().from(technicianSkills);
    const skillMap = new Map<string, Array<{ skillId: string; proficiency: number }>>();
    for (const ts of allSkills) {
      if (!skillMap.has(ts.technicianId)) skillMap.set(ts.technicianId, []);
      skillMap.get(ts.technicianId)!.push({
        skillId: ts.skillId,
        proficiency: ts.proficiencyLevel,
      });
    }

    // 3. Get conflicting assignments
    const conflicts = await this.db
      .select({ technicianId: assignments.technicianId })
      .from(assignments)
      .where(
        and(
          not(inArray(assignments.status, ['completed', 'declined'])),
          sql`${assignments.scheduledStart} < ${request.scheduledEnd}`,
          sql`${assignments.scheduledEnd} > ${request.scheduledStart}`,
        )
      );
    const busyIds = new Set(conflicts.map((c) => c.technicianId));

    // 4. Get active assignment counts (workload)
    const workloadCounts = await this.db
      .select({
        technicianId: assignments.technicianId,
        activeCount: sql<number>`count(*)`,
      })
      .from(assignments)
      .where(inArray(assignments.status, ['assigned', 'accepted', 'in_progress']))
      .groupBy(assignments.technicianId);
    const workloadMap = new Map(workloadCounts.map((w) => [w.technicianId, w.activeCount]));

    // 5. Score each technician
    const scored: ScoredTechnician[] = [];

    for (const tech of allTechs) {
      // Skip if busy during the window
      if (busyIds.has(tech.id)) continue;

      // Skill score (0-1)
      const techSkills = skillMap.get(tech.id) ?? [];
      let skillScore = 0;
      if (request.requiredSkills.length === 0) {
        skillScore = 0.5; // no requirements = neutral
      } else {
        let matched = 0;
        let profSum = 0;
        for (const reqSkill of request.requiredSkills) {
          const found = techSkills.find((s) => s.skillId === reqSkill);
          if (found) {
            matched++;
            profSum += found.proficiency / 3; // normalize to 0-1
          }
        }
        skillScore = request.requiredSkills.length > 0
          ? (matched / request.requiredSkills.length) * 0.6 + (profSum / request.requiredSkills.length) * 0.4
          : 0.5;
      }

      // Workload score (0-1, fewer = better)
      const activeCount = workloadMap.get(tech.id) ?? 0;
      const workloadScore = Math.max(0, 1 - activeCount * 0.2);

      // Proximity score (0-1)
      const proximityScore = tech.defaultLocationId === request.assetLocationId ? 1.0 : 0.3;

      // Availability score: 1.0 since we already filtered busy techs
      const availabilityScore = 1.0;

      // Weighted total
      const weights = { skill: 0.4, availability: 0.3, workload: 0.2, proximity: 0.1 };
      const score =
        skillScore * weights.skill +
        availabilityScore * weights.availability +
        workloadScore * weights.workload +
        proximityScore * weights.proximity;

      scored.push({
        technicianId: tech.id,
        firstName: tech.firstName,
        lastName: tech.lastName,
        score,
        breakdown: {
          skillScore,
          availabilityScore,
          workloadScore,
          proximityScore,
        },
      });
    }

    // Sort by score descending
    scored.sort((a, b) => b.score - a.score);
    return scored;
  }

  async autoAssign(request: SchedulingRequest): Promise<ScoredTechnician | null> {
    const candidates = await this.rankCandidates(request);
    if (candidates.length === 0) return null;

    // For critical work orders, require a minimum score
    const minScore = request.priority === 'critical' ? 0.7 : 0.4;
    const best = candidates[0];

    if (best.score < minScore) return null;
    return best;
  }
}
```

**Evolution path:** When you outgrow greedy scoring, move to constraint programming with a library like `optaplanner` (via JVM sidecar) or a Python CP-SAT solver. The `SchedulingEngine` interface stays the same — only the internals change.

## Step 7: Cross-Database Engineering Decisions

### Why Drizzle ORM (Confirmed)

Drizzle is the right choice for ARUS. It has first-class support for both SQLite (`better-sqlite3`, `libsql`) and PostgreSQL (`node-postgres`, `postgres.js`). The schema-as-code approach means you define schemas per dialect but share the TypeScript types. Migration tooling (`drizzle-kit`) generates dialect-specific SQL.

**Do not use Prisma.** Prisma’s SQLite support has sharp edges around transactions, JSON, and generated columns. Its query engine adds overhead you don’t need on vessel hardware.

**Do not use Kysely unless** you need raw SQL escape hatches — Drizzle covers that with `sql` template literals.

### Transaction Handling

```typescript
// SQLite: transactions are serialized (single-writer). Fast but blocks.
// PostgreSQL: transactions are concurrent with MVCC.
//
// Drizzle's `db.transaction()` works identically for both.
// The key difference: SQLite will SQLITE_BUSY under concurrent writes.
//
// Mitigation for SQLite:
// 1. Use WAL mode (set in client.ts)
// 2. Set busy_timeout (set in client.ts)
// 3. Keep transactions short
// 4. Queue writes in the agent/background jobs

await db.transaction(async (tx) => {
  // This works on both SQLite and PostgreSQL
  const [stock] = await tx.select().from(inventoryStock).where(eq(inventoryStock.id, stockId));
  await tx.update(inventoryStock).set({ quantityReserved: stock.quantityReserved + qty });
  await tx.insert(inventoryReservations).values({ ... });
});
```

### Concurrency + Write Throughput

SQLite is single-writer. This is fine for a vessel with one user instance. It becomes a bottleneck when:

- Multiple users are writing simultaneously (10+ concurrent write transactions)
- Background jobs compete with API writes (prediction batch + user edits)
- Telemetry ingestion rate exceeds ~1000 rows/second

**Mitigations while on SQLite:**

- WAL mode (already set)
- Write queue for background jobs (process sequentially)
- Batch inserts for telemetry (insert 100 rows per transaction, not 1)
- Read replicas aren’t possible — but reads are fast and non-blocking in WAL

### When to Stay on SQLite

- Single vessel deployment (Tauri)
- Development and testing
- Fewer than 5 concurrent users
- Telemetry under 1000 readings/minute
- Total database under 10GB

### When to Move to PostgreSQL

- Cloud deployment serving multiple vessels
- More than 10 concurrent users
- Need for `jsonb` queries, full-text search, or `LISTEN/NOTIFY`
- Telemetry volume over 10K readings/minute
- Need for row-level security or connection pooling

### Migration Path: SQLite → PostgreSQL

1. **Already done right:** Text IDs, TEXT timestamps, TEXT enums, TEXT JSON — all work on both.
1. **Create PostgreSQL schema files** alongside SQLite ones (Drizzle supports this).
1. **Use `pg_dump`-compatible export**: Write a migration script that reads SQLite and bulk-inserts into PostgreSQL.
1. **Dual-run test**: Point staging at PostgreSQL while production stays on SQLite. Run integration tests against both.
1. **Switch**: Update `DB_DIALECT` env var. Deploy.

The repository layer already abstracts Drizzle calls, so no business logic changes.

## Step 9: Portability Rules

### ARUS Cross-Database Engineering Rules

1. **IDs: Always TEXT.** Generate with `nanoid()` or `cuid2()` in the app layer. Never use `SERIAL`, `AUTOINCREMENT`, or database-generated IDs.
1. **Timestamps: Always TEXT in ISO-8601.** Use `new Date().toISOString()`. Never use database-native `TIMESTAMP`, `DATETIME`, or `NOW()`.
1. **Enums: Always TEXT.** Validate in the app with zod. Never use `CREATE TYPE ... AS ENUM` or `CHECK` constraints for enum values.
1. **Booleans: Always INTEGER (0/1) for SQLite.** Drizzle’s PostgreSQL schema can use native `boolean`. Keep the app-layer type as `boolean` and let Drizzle handle conversion.
1. **JSON: Always TEXT.** `JSON.stringify()` on write, `JSON.parse()` on read. Never depend on `jsonb` operators in queries. If you later need `jsonb` queries on PostgreSQL, add them as optional optimizations behind a dialect check.
1. **Avoid these SQL features:** CTEs with `RECURSIVE` (different behavior), window functions (limited in older SQLite), `RETURNING *` (use `RETURNING` with explicit columns), `ON CONFLICT ... DO UPDATE` (syntax differs slightly), `ARRAY` types, `LATERAL` joins.
1. **Safe SQL features (both dialects):** `JOIN`, `LEFT JOIN`, `GROUP BY`, `HAVING`, `ORDER BY`, `LIMIT`, `OFFSET`, `UNION ALL`, `CASE WHEN`, `COALESCE`, `COUNT/SUM/AVG/MIN/MAX`, `LIKE`, `IN`, `BETWEEN`, `EXISTS`, basic `INSERT/UPDATE/DELETE`, `CREATE INDEX`.
1. **Migrations: Separate directories.** `drizzle/sqlite/` and `drizzle/pg/`. Generate with `drizzle-kit generate` per dialect. Never hand-edit migration SQL unless you edit both.
1. **Repository pattern: Always use Drizzle’s query builder.** Never write raw SQL strings in service or route files. Raw SQL is allowed only inside repository files and must be tested against both dialects.
1. **Testing: Run integration tests against both.** Use SQLite for fast unit/integration tests. Run the full suite against PostgreSQL in CI before any release that will touch cloud deployment.
1. **No database-specific triggers or stored procedures.** All business logic lives in TypeScript services. This is non-negotiable.
1. **Transactions: Keep them short.** Under 50ms for SQLite. No network calls inside transactions.

- TEXT IDs (nanoid), TEXT timestamps (ISO-8601), TEXT enums (zod-validated), TEXT JSON (stringify/parse)
- Drizzle ORM with separate SQLite/PG migration directories
- No database triggers, stored procedures, or dialect-specific types in the core schema
- Repository pattern isolates all SQL; services are database-agnostic
- WAL mode + busy_timeout for SQLite concurrency; swap to PG connection pool when ready

### Migration Path: SQLite → PostgreSQL

1. Your schemas already use portable types — no rewrite needed
1. Create `drizzle/pg/` migration directory, generate from same schema definitions (with PG-specific column types where Drizzle handles the mapping)
1. Write a one-time data migration script: read SQLite → bulk insert PostgreSQL
1. Dual-run integration tests against both
1. Flip `DB_DIALECT=postgresql` in production config
1. Optional: add `jsonb` columns, full-text search, and `LISTEN/NOTIFY` as PostgreSQL-only enhancements after migration
