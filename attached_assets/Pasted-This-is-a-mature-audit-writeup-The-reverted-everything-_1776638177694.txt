This is a mature audit writeup. The “reverted everything, shipped the document instead” outcome is genuinely the right call here — that takes discipline on a long codebase session. A few reactions worth noting:
The 9% false-positive rate from knip is the headline finding and you buried it. Static tools in a codebase with three dynamic-import patterns don’t give 9% bad signal — they give 9% dangerous signal. Every one of those 85 files was load-bearing. Consider moving this up into the executive summary or a separate “Don’t Trust Knip Here” callout. Future-you or a teammate skimming this doc needs to see that number before they touch anything.
Phase 1 item 3 is the most valuable artifact. Documenting the dynamic-loader pattern in replit.md has compounding returns every time someone new approaches this work. Everything else in Phase 1 is defensive; that one is educational. Prioritize writing it while the cascade is fresh — the specific shapes of the three patterns and which tools miss which. “Compiled-path imports like ./foo.js from TS” is exactly the kind of detail that disappears from memory in two weeks.
On Phase 2’s ordering: “Test fixtures and .bak files” should be first, not third. They’re the lowest-risk and will build confidence in the safety net before you do anything that could break production. Client components are a better second because the blast radius is still contained to the UI. Server barrel re-exports go last — an export’s absence only surfaces when something imports it, and observability code often gets imported from unexpected places (logging wrappers, error handlers, middleware).
The duplicate-exports finding is worth separating. “42 duplicate exports — these are name collisions, not dead code” is a genuinely different problem from dead code. It probably indicates either barrel-file drift or copy-paste module creation. Worth its own short audit — each collision is a judgment call about which is canonical.
One caution about the ml-data-preprocessing chain. You have two reasonable-sounding options: restore the module or remove the reference. Before picking, find out whether predictors.ts is reachable at runtime. If no code path actually calls it in production, both options are fine and “remove the reference” is cheaper. If it is reachable, “restore the module” is the only safe answer and you need to understand what was in it. Don’t let this become a coin-flip decision — one of the two options might be silently breaking a feature in prod.
A small process suggestion for Phase 2. Before each per-domain batch, capture a pre-deletion snapshot: running process list of domain-router-registry loaders, domain registration counts, and a successful test run. Diff against post-deletion. The asymmetry of the current safety net is that boot-test catches catastrophic failures but misses features that were lazily loaded and are now silently unreachable because the user hasn’t clicked the right button yet. Manual smoke-testing a domain’s primary user flow after deleting from it is the only way to catch that class of bug, and it’s worth the 5 minutes per domain.
The overall direction is sound. The instinct to back out, document, and build the safety net before doing anything else is the right call. Ship the audit doc, then do Phase 1 in order, then come back for Phase 2 only after the CI boot check is in place.​​​​​​​​​​​​​​​​
