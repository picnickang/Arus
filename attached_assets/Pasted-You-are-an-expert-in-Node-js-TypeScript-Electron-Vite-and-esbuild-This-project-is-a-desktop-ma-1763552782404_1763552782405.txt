TASK 1 – Fix fs.existsSync static file bug
========================

- Search the codebase (especially in server/index.ts and related files) for the static-serving logic that prints:

  "Checking:" followed by the client directory path
  and where fs.existsSync(...) is used.

- You will see code that has been bundled to something like `fs18.existsSync`, producing:

  "fs18.existsSync is not a function"

- This usually means the source did something like:

  import fs from 'fs/promises';
  // or import * as fs from 'fs/promises';

  if (fs.existsSync(somePath)) { ... }

  which is incorrect, because fs.promises does not include existsSync.

- Fix this in the ORIGINAL TypeScript / source files (not directly in the bundle) by:

  - Importing the correct module for sync checks:

    import fs from 'fs';
    // or: import * as fs from 'fs';

  - OR by using fs.promises with an async access check instead of existsSync, e.g.:

    import fs from 'fs/promises';

    async function pathExists(p: string): Promise<boolean> {
      try {
        await fs.access(p);
        return true;
      } catch {
        return false;
      }
    }

    if (await pathExists(clientDir)) { ... }

- Choose the simplest approach: for this project, it’s acceptable to use sync `fs.existsSync` at startup. So please:

  - Use `import fs from 'fs';` (or `import * as fs from 'fs';`)
  - Call `fs.existsSync(...)` on that sync fs object.
  - Make sure no `fs.promises` object is mistakenly used with existsSync anywhere.

- Rebuild the server bundle, then ensure that:

  - No code path still references `fsXX.existsSync` where fsXX is a promises object.
  - The static-serving path does NOT throw, and the app does not log "fs18.existsSync is not a function" anymore.
