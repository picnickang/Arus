REPLIT AI PROMPT — Auto-Assess PdM UX/UI Feature Fit + POCs + Screenshots
(Stack: Express.js + React 18 + Wouter + TanStack Query)

CONTEXT
Monorepo layout assumption:
- server/ (Express API: TF.js/ONNX PdM)
- client/ (React 18 + Wouter + TanStack Query)
Goal: AUTOMATE assessment & evidence: query real endpoints, score the proposed features, build minimal POCs (behind a flag) for the top candidates, and capture screenshots for review. Do NOT refactor production code. POCs must live under /poc and be disabled unless explicitly enabled.

FEATURES TO SCORE
F1 Fleet Overview (cards + RAG + filters + sort)
F2 Vessel Performance (4 VPS charts: Load↔SFOC, Fuel↔Time, Power↔STW, Load Histogram)
F3 Anomaly Overlays (markers + threshold bands + “view alert”)
F4 Context Layer (CLE) w/ AIS & Metocean (mode/sea-state bands)
F5 Alerts → Ack/WO
F6 Maintenance list/timeline
F7 Baselines & Fleet benchmarking (baseline + fleet avg + percentiles)
F8 Compliance/CII mini-tile + report stubs
F9 Settings/Customization (toggle, thresholds, units)
F10 Narrative “Performance Summary” card

A) DEPENDENCY CHECK (scripts)
Create `scripts/audit_stack.js` (Node) that:
- Confirms Express present (`server/index.js` or `server/app.js`) and routes under `server/routes/**`.
- Confirms React 18 + Wouter + TanStack Query: look for `client/src/main.(t|j)sx`, `"wouter"`, `"@tanstack/react-query"` in client package.json or imports.
- Prints detected versions and paths.

Create `scripts/ping_endpoints.js` that tries these (dry run; tolerate 404/500 but record):
- GET /api/fleet/summary
- GET /api/vessels/V001/performance
- GET /api/vessels/V001/baselines
- GET /api/vessels/V001/overlays
- GET /api/vessels/V001/context
- GET /api/vessels/V001/alerts
If unknown vessel id, also try the first id from /api/fleet/summary. Validate basic shape (arrays aligned; presence of fields). Output JSON to `tmp/endpoint_report.json`.

B) SCORING (scripts)
Create `scripts/score_features.js` that:
- Reads `tmp/endpoint_report.json`.
- For F1..F10 computes:
VALUE (1–5), EFFORT (1–5), DATA FIT (1–5), UX COMPLEX (1–5), RISK (1–5).
Suggested heuristics:
• F1 needs /api/fleet/summary (if 200 + ≥1 item → DataFit≥3)
• F2 needs /performance arrays & segments aligned (equal lengths → DataFit≥3; has baselines → +1)
• F3 needs /overlays points → DataFit baseline 2; if includes context fields → +1
• F4 needs /context; if AIS/metocean fields present → +1
• F5 needs /alerts with ack flags or recommendation → +1
• F6 can run with empty list (mockable) → DataFit≥2 if /alerts exists
• F7 needs /baselines + fleet_avg fields → DataFit≥3
• F8 needs any emissions/CII field (else DataFit 1, mockable)
• F9 purely UI (mockable) → DataFit≥2 if perf/overlays exist
• F10 needs perf + baselines; if contributors present → +1
- OverallFit = round((VALUE*1.5 + DATA FIT + (6 - RISK)) - (EFFORT/2 + UX COMPLEX/2)).
- Writes:
- `docs/assessment_baseline.md` (raw endpoint findings)
- `docs/assessment_scores.md` (scoring matrix with justifications)
- `docs/assessment_summary.md` (top Adopt/Pilot/Defer, dependency gaps, next steps)
- Prints compact table to console.

C) GATED POCs (only for top 2 features with OverallFit ≥ 6)
ENV FLAG: `UI_POC=1`

Client side (React + Wouter):
- Create `client/src/poc/AppPoc.tsx` and `client/src/poc/routes.tsx` with declarative routes:
`/poc/fleet` and `/poc/performance`.
- `/poc/fleet`: simple table or cards using TanStack Query to call `/api/fleet/summary`; sortable by anomaly_score; RAG pill.
- `/poc/performance`: ONE chart (Fuel vs Time) using Recharts/visx; overlay 1–2 markers from `/api/vessels/:id/overlays` if available; time range dropdown mocked.
- Add `client/src/poc/useApi.ts` wrapper around TanStack Query with baseURL.
- Update Wouter mounting in `client/src/main.(t|j)sx`:
If `import.meta.env.VITE_UI_POC==='1'` (or process.env.UI_POC for CRA), register POC routes under a separate `<Switch>` without touching prod routes.

Server side (only if endpoints missing):
- Create `server/routes/poc.js` with minimal mock endpoints `/api/poc/fleet` and `/api/poc/performance` returning fixed arrays. Only mount this router if `process.env.UI_POC==='1'`.

D) SCREENSHOTS (Puppeteer)
Add dev dependency: `puppeteer`.
Create `scripts/screenshot_poc.js` that:
- Assumes `server` on `http://localhost:3000` (Express) and `client` dev server on `http://localhost:5173` (Vite) — adjust if needed.
- Opens:
- `http://localhost:5173/poc/fleet`
- `http://localhost:5173/poc/performance?vessel=V001`
- Waits for key selectors (e.g., `.poc-ready` class on each page) then saves:
- `docs/screens/poc_fleet.png`
- `docs/screens/poc_performance.png`
If endpoint data is missing, it still shoots the POC with mocked data.

E) NPM SCRIPTS (root and packages)
Root `package.json`:
- `"audit:stack": "node scripts/audit_stack.js"`
- `"audit:endpoints": "node scripts/ping_endpoints.js"`
- `"score:features": "node scripts/score_features.js"`
- `"poc:screens": "node scripts/screenshot_poc.js"`
- `"assess:all": "npm run audit:stack && npm run audit:endpoints && npm run score:features"`

Client `package.json`:
- Ensure `"@tanstack/react-query"` and `"wouter"` present.
- Add `"dev:poc": "cross-env VITE_UI_POC=1 vite"` (or CRA equivalent).

Server `package.json`:
- Add `"dev": "nodemon server/index.js"` (if not present).

F) OUTPUT REQUIREMENTS
- Console: table of OverallFit by feature, top 2 chosen for POCs.
- Files generated:
- `docs/assessment_baseline.md`
- `docs/assessment_scores.md`
- `docs/assessment_summary.md`
- `docs/screens/poc_fleet.png`, `docs/screens/poc_performance.png` (if UI_POC=1 and Puppeteer run succeeded)
- Each doc must include a “Next Steps” checklist with exact endpoints/components to finalize.

G) RUNBOOK (print at end)
1) Install deps if needed: `npm i puppeteer cross-env` (and `nodemon` if missing)
2) Start backend: `npm --prefix server run dev`
3) Start frontend (POC mode): `npm --prefix client run dev:poc`
4) In another shell:
- `npm run assess:all` (runs stack audit → endpoint ping → scoring)
- `npm run poc:screens` (captures screenshots)
5) Review:
- open `docs/assessment_summary.md`
- inspect screenshots in `docs/screens/`
- visit `/poc/fleet` and `/poc/performance` live

SCORING GUIDANCE (baked in code)
- Treat aligned arrays + segments presence as a strong signal F2 is viable.
- If overlays include `context.mode` or `context.sea_state`, boost F3 & F4.
- Penalize features that require new backends you don’t have (e.g., CII) unless mockable without schema churn.
- Prefer features improving ops decisions within 1–2 screens (F1–F3 first).

DO NOT
- Modify production routes/components beyond read-only inspection.
- Add heavyweight dependencies for charts; reuse existing chart lib if present, else minimal Recharts.

DELIVERABLE
At completion:
- A scored, evidence-backed recommendation of which features to adopt/pilot/defer.
- Minimal, isolated POC pages under `/poc` you can show to stakeholders—with screenshots saved to docs.
