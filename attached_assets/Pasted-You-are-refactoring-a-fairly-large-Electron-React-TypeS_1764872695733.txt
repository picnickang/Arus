Refactor goals



For each “Needs modularization” file:

Identify responsibilities

Break down what the file does into:

Routing / page layout / tab structure

Data fetching & mutations (React Query, API calls)

Domain logic (calculations, transforms, validation)

Reusable UI elements (tables, grids, dialogs, forms, filters)

Cross-cutting helpers (formatters, mappers, constants)

Introduce a consistent domain structure

Prefer a domain-based layout such as:

src/features/engine-logbook/

components/…

hooks/…

lib/…

src/features/deck-logbook/…

src/features/crew/…

src/features/inventory/…

src/features/email-alerts/…

src/features/pdm/…

src/features/governance/…

When splitting:

Move data-fetching logic into useXxx hooks (e.g. useEngineLogbook, useCrewScheduler, useInventoryItems).

Move repeated layout fragments (tables, rows, dialogs, filter bars) into components/.

Move calculations, mappers, anomaly detection, and schema conversion into lib/ or utils/.

Keep all external contracts identical

Keep route paths, query keys, API URLs, HTTP payload shapes, and query parameters the same.

Keep React component public props the same for anything exported outside its domain.

Keep data-test IDs (data-testid) unchanged.

Keep translation keys, labels, and copy the same.

Do not change business logic semantics (especially for compliance, alerts, crew hours, engine log calculations, etc.).
