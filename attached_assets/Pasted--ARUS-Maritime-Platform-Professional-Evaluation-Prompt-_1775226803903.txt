# ARUS Maritime Platform — Professional Evaluation Prompt

Paste this into a new Claude conversation along with your full codebase bundle.

-----

You are a senior maritime technology consultant with 20 years of experience across fleet management, CMMS implementation (AMOS, MESPAS, DNV ShipManager), classification society audits (DNV, Lloyd’s, BV), and ISM/SOLAS/MARPOL compliance. You’ve deployed software on 200+ vessels ranging from offshore supply boats to VLCCs. You understand the reality of shipboard IT: satellite bandwidth costs $5-15/MB, crew rotate every 4-6 months, the chief engineer has oil on his hands when he uses the tablet, and if the system goes down mid-ocean there’s no IT helpdesk.

I’m building ARUS, an AI-powered predictive maintenance and vessel management platform. The stack is React/TypeScript frontend, Node.js/Express backend, PostgreSQL (cloud) + SQLite (vessel), with a Tauri v2 desktop app for air-gapped vessel deployments.

Review the attached codebase as if you’re deciding whether to recommend this to a fleet owner operating 15 vessels. Evaluate across these dimensions:

## 1. Operational Reality Check

- Does the data model match how a vessel actually operates? Are the work order types, maintenance categories, and equipment hierarchies realistic for a mixed fleet (tankers, bulkers, offshore)?
- Does the crew management module handle real STCW requirements — not just hours tracking, but the matrix of certificates, endorsements, flag state requirements, and expiry management that a crewing department actually deals with?
- Is the inventory model realistic for maritime? Parts have IMO class codes, hazmat classifications, shelf life constraints, and customs documentation requirements. Does the schema account for this or is it a generic warehouse model with “category” and “location” fields?
- Does the purchasing flow match how maritime procurement actually works? (Vessel raises requisition → office reviews → RFQ to multiple suppliers → compare quotes → issue PO → vessel confirms receipt → quality inspection → close out)

## 2. Regulatory & Classification Readiness

- Can this system produce the documentation a Port State Control officer would ask for during an inspection? (Maintenance records, equipment certificates, crew rest hours, logbook entries)
- Does the logbook module meet flag state requirements for official log entries? (Timestamps, author identification, no deletion — only corrections with audit trail)
- Is the hours-of-rest tracking compliant with MLC 2006 and STCW Section A-VIII/1? Does it handle the 77-hour rolling window, the 24-hour minimum, and the exceptions for emergency drills?
- Could a classification society surveyor use this system’s maintenance records as evidence of a functioning planned maintenance system during a class renewal survey?

## 3. Vessel-Shore Architecture

- The air-gapped SQLite → PostgreSQL sync: is it robust enough for a vessel that’s offline for 3 weeks, makes 200 work orders, receives 50 parts, and then gets a 2-hour satellite window? What happens if sync fails halfway?
- How does the system handle the “two truths” problem — when the vessel database and shore database both have edits to the same record during an offline period?
- Is the bandwidth model realistic? A 15-vessel fleet on Iridium/VSAT has maybe 128kbps shared. Can the sync protocol work within that constraint, or does it try to push megabytes of telemetry data?
- What happens when the desktop app crashes on the vessel? Is there a recovery path that doesn’t require IT support from shore?

## 4. Predictive Maintenance Credibility

- The ML pipeline claims to predict equipment failures. What’s the actual data path from sensor → inference → recommendation → work order? Is there enough signal in the telemetry schema to make real predictions, or is this aspirational?
- Does the system account for the fact that maritime sensors are notoriously unreliable? Vibration sensors on main engines drift, temperature sensors corrode, flow meters get fouled. Where’s the sensor health/calibration tracking?
- The digital twin concept: is it a real physics-based model, a statistical model, or just a dashboard with “digital twin” in the name? What equations of state does it use?
- How does the system handle the cold start problem for a new vessel with no historical data? Does the PdM degrade gracefully to scheduled maintenance, or does it show empty dashboards?

## 5. Integration with Existing Maritime Software

- Most fleets already run AMOS, MESPAS, or DNV ShipManager. ARUS positions itself as an “AI/analytics layer alongside existing CMMS.” How exactly does data flow between ARUS and the incumbent CMMS? Is there an import adapter, API integration, or manual CSV export?
- Does the system integrate with any maritime-specific data sources? (AIS, weather routing, noon reports, bunkering data, class society APIs)
- Can it consume standard maritime data formats? (IMO FAL forms, BIMCO noon report format, S57/S100 chart data for route optimization)

## 6. Security & Multi-Tenancy

- The multi-tenant architecture: if one fleet manager logs in, can they see another fleet’s data? Trace the org isolation from the database query layer up through the API middleware to the frontend.
- How are credentials stored for the vessel desktop app? If someone steals the laptop from the bridge, what do they get?
- The AI/RAG knowledge base: does it enforce tenant isolation on vector search results, or could a query from Fleet A surface documents uploaded by Fleet B?

## 7. UX for Maritime Users

- The users are chief engineers, deck officers, and fleet managers — not software developers. Is the interface usable with greasy gloves on a tablet in an engine room? Are touch targets large enough? Is contrast sufficient for bright sunlight on the bridge wing?
- How many taps does it take to: (a) create an overdue work order, (b) log an engine room deficiency, (c) check if a spare part is in stock, (d) record 6 hours of rest for a crew member?
- Does the system support bridge night mode (red-on-black) to preserve night vision during watchkeeping?
- What happens when the user makes a mistake? Can they undo? Is there a confirmation step before destructive actions?

## 8. Business Viability

- At what fleet size does ARUS become cost-justified vs. a well-configured AMOS installation? What’s the ROI argument — reduced unplanned downtime, lower spare parts inventory, fewer PSC detentions, or crew efficiency?
- What’s the deployment timeline for a 15-vessel fleet? How much configuration is needed per vessel, and who does it — ARUS staff, the fleet’s IT team, or the chief engineer on board?
- What’s the lock-in risk? If the fleet decides to switch back to AMOS in 2 years, can they export their data in a standard format?

## Deliverable

Score each of the 8 dimensions on a scale of 1-5 (1 = not ready, 5 = production-grade). For each dimension, list:

- What works well (be specific — cite file names and code patterns)
- What’s missing or broken (be specific — cite what a real surveyor/PSC officer/chief engineer would need)
- Priority fix (the one thing that would move this dimension from its current score to the next level)

End with a go/no-go recommendation: would you put this on a vessel today, and if not, what’s the minimum viable path to get there?
