You are a senior QA engineer and staff software engineer validating ARUS after a major convergence and architecture-hardening effort.

Your job is to create and run a COMPLETE test plan that proves:
1. functionality still works
2. critical workflows still work
3. CRUD still works across major domains
4. multi-tenant safety still works
5. dual-database behavior is still consistent
6. architecture guardrails still hold
7. no legacy fallbacks or hidden regressions remain

IMPORTANT RULES
- Do not just describe tests. Implement them where missing and run what can be run.
- Prefer real integration/API/E2E tests over shallow mocks.
- Test behavior through the real app layers whenever practical.
- Preserve current functionality.
- If a test cannot be run, explain exactly why and what is missing.
- At the end, provide a pass/fail report with evidence.

==================================================
PHASE 1 — DISCOVER CURRENT TEST SURFACE
==================================================

1. Inspect the repo and list:
- existing unit tests
- integration tests
- API tests
- E2E tests
- test utilities / fixtures
- CI scripts
- guardrail scripts
- build/test commands

2. Identify gaps between current tests and the required validation below.

3. Do not stop at analysis. Add missing tests.

==================================================
PHASE 2 — REQUIRED TEST CATEGORIES
==================================================

Create and/or run tests for all of the following.

--------------------------------------------------
A. ARCHITECTURE / GUARDRAIL TESTS
--------------------------------------------------

Run and verify:
- npm run check:schema
- npm run check:storage-imports
- npm run check:schema-imports
- npm run check:domain-boundaries
- npm run check:route-registration
- npm run check:guards

Validate:
- no forbidden storage facade imports
- no forbidden schema imports
- no forbidden cross-domain imports
- no invalid route registration patterns
- no blocking dual-db drift
- no missing runtime schema exports

Add tests or script improvements if any guardrail is missing or too weak.

--------------------------------------------------
B. BUILD / STARTUP / SMOKE TESTS
--------------------------------------------------

Run:
- install
- typecheck
- build
- server startup
- client startup if applicable

Verify:
- app boots cleanly
- no missing imports
- no route registration failures
- no schema/runtime failures
- no obvious console/runtime startup errors

--------------------------------------------------
C. CRITICAL ROLE-BASED WORKFLOW TESTS (E2E / integration)
--------------------------------------------------

Create or run end-to-end tests for the most important real workflows.

1. Vessel Engineer workflow
- open dashboard
- review active alerts/issues
- open equipment
- create work order
- update work order
- complete work order
- verify result reflected in UI/system

2. Fleet Manager workflow
- open fleet overview
- identify vessel needing attention
- drill into vessel
- inspect maintenance backlog
- assign/update work order
- verify list/status changes

3. PdM / Reliability Engineer workflow
- open PdM dashboard/platform
- review anomaly/prediction
- inspect contributing factors or details
- take or review recommended action
- confirm resulting data/record flow remains valid

4. Sync / Admin workflow
- open sync/admin status screens
- verify sync status/health loads
- confirm no broken legacy fallback behavior
- confirm current canonical endpoints are used

5. Navigation workflow
- main navigation routes
- deep links
- bookmarks / direct URL entry for important pages
- ensure canonical routes resolve correctly
- ensure no broken internal links

--------------------------------------------------
D. WORK ORDER FLOW TESTS (HIGHEST PRIORITY)
--------------------------------------------------

Create deep tests for work orders because this domain was heavily refactored.

Test:
- create work order
- read work order
- update work order
- open drawer/details
- complete through canonical completion path
- verify completion endpoint behavior
- verify feedback persistence if applicable
- verify UI updates after completion
- verify list/detail/dashboard consistency

Edge cases:
- invalid completion payload
- non-existent work order
- unauthorized org access
- double submit / duplicate completion attempt
- refresh during workflow
- cancel out of completion flow
- offline/online behavior if supported

Confirm:
- legacy completion fallback is not used
- canonical completion path is the only active path in production workflow

--------------------------------------------------
E. FULL CRUD TESTS BY DOMAIN
--------------------------------------------------

For each major domain, implement API/integration CRUD tests through real application layers, not raw DB-only tests.

For each domain test:
- Create
- Read
- Update
- Delete / archive / complete / deactivate (whatever is correct for that domain)
- list/filter/pagination where relevant
- invalid payload handling
- auth handling
- tenant isolation
- PostgreSQL behavior
- SQLite behavior if supported

Priority domains:
1. Work Orders
2. Equipment
3. Inventory / Parts / Service Orders if present
4. PdM / Predictions / Anomalies
5. Users / org-scoped admin entities if applicable
6. Sync records or jobs if CRUD-like behavior exists

For each CRUD endpoint validate:
- required fields enforced
- bad types rejected
- unknown or forbidden fields handled correctly
- partial update behavior correct
- delete/close/archive behavior correct
- references/relations behave correctly

--------------------------------------------------
F. API CONTRACT TESTS
--------------------------------------------------

For major endpoints, test:
- valid request returns correct response shape
- invalid request rejected with correct status
- missing required fields rejected
- malformed IDs rejected
- unauthorized request rejected
- wrong-tenant request blocked
- deprecated/removed endpoints are not silently still active unless intentionally retained

Focus especially on:
- work orders
- PdM
- sync
- equipment
- inventory/service orders
- dashboard/status summary APIs

--------------------------------------------------
G. TENANT ISOLATION / AUTH TESTS
--------------------------------------------------

For each critical domain:
- create data in org A
- verify org A can access it
- verify org B cannot read/update/delete it
- verify list endpoints do not leak cross-tenant records
- verify auth failures are handled correctly

These tests are mandatory.

--------------------------------------------------
H. DUAL-DATABASE CONSISTENCY TESTS
--------------------------------------------------

Run a meaningful subset of the same tests in both:
- PostgreSQL mode
- SQLite/offline mode

At minimum verify in both modes:
- startup works
- migrations/schema runtime works
- work order CRUD works
- equipment CRUD works
- PdM read/list/detail workflows work
- sync/status behavior works where applicable
- validation and error semantics are consistent

If any behavior differs intentionally, document it explicitly.

--------------------------------------------------
I. REDIRECT / ROUTE REGRESSION TESTS
--------------------------------------------------

Because redirects and route migrations were cleaned up, verify:
- canonical routes work
- remaining redirects point to correct canonical targets
- removed routes are not still referenced internally
- no internal navigation depends on legacy redirects
- important external/deep links still behave as intended

--------------------------------------------------
J. UI / INTERACTION REGRESSION TESTS
--------------------------------------------------

Verify:
- drawers open/close correctly
- modals open/close correctly
- FAB buttons do not overlap
- Copilot / floating actions do not block critical UI
- forms submit correctly
- validation messaging appears correctly
- mobile/responsive behavior is still usable on key screens

Focus on:
- home/dashboard
- work orders
- PdM
- sync/admin
- any page with drawers, banners, FABs, or overlapping controls

--------------------------------------------------
K. PERFORMANCE / SMOKE SANITY TESTS
--------------------------------------------------

Run lightweight performance sanity checks:
- dashboard load
- work order list load
- PdM page load
- sync/admin page load

Do not do deep benchmarking unless tooling already exists.
Just catch obvious regressions or broken loading behavior.

--------------------------------------------------
L. REGRESSION SEARCHES / STATIC VERIFICATION
--------------------------------------------------

Search the repo for:
- old completion endpoints
- old fallback endpoints
- forbidden route patterns
- forbidden storage facade imports
- stale legacy redirects
- TODO/FIXME/deprecated markers in critical paths

Report anything that still looks risky or inconsistent.

==================================================
PHASE 3 — IMPLEMENTATION GUIDANCE
==================================================

When writing tests:
- use the repo’s existing test framework if present
- prefer API/integration tests for CRUD and contracts
- prefer E2E tests for workflows and navigation
- avoid excessive mocking
- use reusable fixtures/factories for:

  - work orders
  - predictions/anomalies
  - inventory/service orders if relevant

Where appropriate, add:
- test helpers for authenticated org-scoped requests
- test helpers for switching DB mode
- canonical route assertions
- response-shape assertions
