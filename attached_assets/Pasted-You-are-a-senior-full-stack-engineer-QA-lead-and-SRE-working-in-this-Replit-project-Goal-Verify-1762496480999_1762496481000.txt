### 3) Core CRUD forms — **Parts / Vessels / Equipment / Sensors**
Implement or fix the following user flows **end-to-end** (client → API → DB → UI refresh):

(A) Parts
- Form fields: vendor, part name, SKU/code, unitCost, unit, reorderLevel, notes.
- Validate: required fields, numeric ranges, duplicate SKU protection per org.
- Actions: create, update, delete; stock linkage maintained.
- Sorting/filtering: vendor (SORTABLE), name, cost, updatedAt.
- Side effects: creating part does not silently create stock; reservation checks use partId.

(B) Vessels
- Fields: name, IMO/registration, status, location, notes.
- Unique (name, org) rule; soft delete supported.
- On delete: protect if equipment still attached.

(C) Equipment
- Fields: equipment type, vesselId, manufacturer/model, installDate, status.
- On create/update: auto-invoke EquipmentAnalyticsService to create default **sensor configs** & alert configs for that type (engine, pump, compressor, generator, hvac, boiler).
- Sensor coverage helper surfaces missing sensors in UI.

(D) Sensors
- Fields: equipmentId, sensorType, unit, sampleRateHz, thresholds (warnLo/Hi, critLo/Hi), enabled.
- Validate sampleRate bounds, threshold ordering, and unit compatibility.
- On enable/disable: recalc active thresholds; conflicts prevented (one active per device/sensor).

Implementation notes:
- All forms must use TanStack Query mutations with optimistic updates + server-side validation messages.
- Show toasts for success/error; disable buttons during pending requests.
- Real-time: on create/update/delete, broadcast an event and **invalidate relevant query keys** (use your `useRealtimeSync` hook mapping).

Artifacts:
- Cypress specs: `e2e/forms.parts.cy.ts`, `e2e/forms.vessels.cy.ts`, `e2e/forms.equipment.cy.ts`, `e2e/forms.sensors.cy.ts`
- Unit tests for validators: `client/tests/validators/*.test.ts`
