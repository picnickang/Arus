You are working on ARUS, a fleet operations / predictive maintenance platform built in React + TypeScript.

Your task is to implement a new equipment-centric Equipment Page that becomes the primary hub for a single piece of equipment.

Goal

Replace fragmented tool-centric workflows with one page that shows everything important about a selected equipment item in one place:
	•	health
	•	prediction
	•	operational context
	•	work orders
	•	procurement/service status
	•	diagnostics
	•	activity history
	•	navigation to related tools

This page should feel operational and decision-oriented, not like a technical demo.

⸻

Core UX Rules
	1.	Everything starts from equipment
	•	No raw equipment IDs or twin IDs in the UI
	•	Navigation should preserve equipment context everywhere
	2.	No collapsibles for primary content
	•	If information is important for decision-making, it must be visible
	•	Only secondary detail like contributing signals may be collapsed
	3.	No icon-only critical actions
	•	Buttons must be labeled clearly, e.g.:
	•	Create Work Order
	•	Acknowledge
	•	View Procurement
	4.	Translate technical outputs into operational language
	•	Avoid exposing ML jargon like z-score, kurtosis, normalized drift as primary text
	•	Show operational interpretation first, with technical detail secondary
	5.	This page is the main hub
	•	It should reduce the need to jump between PdM Dashboard, PdM Pack, Digital Twin, and work order/procurement areas

⸻

Build This Page

Create a new page or refactor the current Equipment Intelligence detail experience into a full equipment-centric page with this structure:

1. Header

Show:
	•	equipment name
	•	vessel name
	•	equipment type / system
	•	health
	•	RUL
	•	prediction confidence

Example layout:
	•	Breadcrumb: Vessel → System → Equipment
	•	Title row: Deck Crane — Green Belait
	•	Meta row: Deck Equipment / Crane
	•	Summary chips or metrics:
	•	Health: 34%
	•	RUL: 12 days
	•	Confidence: 87%

2. Sticky Action Bar

Always visible near top while scrolling.

Actions:
	•	Create Work Order
	•	Acknowledge
	•	Assign
	•	More menu for secondary actions

No unlabeled icons for critical actions.

3. Assessment Section

Merge current overview + prediction concepts into one section.

Show:
	•	AI assessment in plain language
	•	recommended action
	•	contributing signals (secondary, can be expandable)
	•	service dates:
	•	last service
	•	next scheduled maintenance

This section should answer:
	•	what is wrong
	•	how urgent is it
	•	what should be done next

4. Operational Context Section

This is critical.

Add mock or real support for:
	•	vessel status (at sea / in port)
	•	next port call
	•	parts availability
	•	maintenance window alignment
	•	operational recommendation

Example:
	•	Next Port: Rotterdam in 10 days
	•	Parts Availability: In stock at Rotterdam
	•	Recommendation: Safe to defer repair until port call

Even if some of this is mock data initially, structure the UI for it now.

5. Needs Action Section

Show top urgent linked items relevant to this equipment or vessel context.

Examples:
	•	overdue work order
	•	critical related alert
	•	crew compliance issue if relevant
	•	related anomaly

This should be a short strip/list of 3–5 items max.

6. Evidence Section

Merge telemetry trend + recent work history into a single evidence section.

Show:
	•	translated trend summary in plain language
	•	sparkline / mini trend chart
	•	key signal changes
	•	recent maintenance/work history

Do not make this overly technical.
Lead with operational interpretation, then evidence.

7. Work + Procurement Section

This is important.

Model:
	•	Work Order is the engineer-owned operational object
	•	Service Order / Procurement object may live separately in procurement
	•	But procurement/service status must be visible here

Show:
	•	linked work orders
	•	linked service/procurement records
	•	procurement status summary:
	•	vendor sourcing
	•	parts ordered
	•	ETA

Example:
	•	WO-2847 — Repair — Waiting for parts
	•	SO-1023 — Vendor sourcing — ETA Apr 12

Buttons:
	•	Create Work Order
	•	View Procurement Details

8. Diagnostics Section

Move PdM Pack-style diagnostics into this equipment page conceptually.

Show cards for:
	•	Bearing Analysis
	•	Pump Analysis
	•	Other relevant diagnostics

Rules:
	•	no manual comma-separated input
	•	use existing telemetry / sensor data if available
	•	diagnostics should persist as history, not disappear after navigation
	•	if full integration is not possible yet, scaffold the structure and use placeholder data wired cleanly for later integration

9. Activity Timeline

Add a unified reverse-chronological activity feed for this equipment.

Include items such as:
	•	telemetry anomaly
	•	prediction updated
	•	work order created / completed
	•	procurement update
	•	diagnostic run completed

This should replace multiple disconnected status tables.

10. Deep Links / Related Tools

At bottom or right rail, add context-preserving navigation:
	•	Digital Twin
	•	Maintenance Schedule
	•	Diagnostics Detail
	•	Fleet Comparison

These links must carry selected equipment context.

⸻

Important Product Decisions

Please implement these assumptions:
	1.	Ask AI should not be a separate tab inside equipment detail
	•	Remove or avoid building a dedicated Ask AI section here
	•	If there is an existing global copilot, assume it can be equipment-scoped later
	•	Do not create another separate chat UI in this page
	2.	Do not require manual ID entry anywhere
	•	Replace any raw ID-based flow with equipment context
	•	Use route params, selected equipment state, or search-based selection
	3.	Keep engineer workflow primary
	•	This page is for decision-making and action
	•	Data science internals should not dominate the UI

⸻

Technical Guidance
	•	Use React + TypeScript
	•	Reuse existing components where sensible, but do not preserve old layout patterns if they create clutter
	•	Prefer visible sections over nested tabs/collapsibles
	•	Avoid card explosion
	•	Keep the layout clean, readable, and operational

Suggested page sections in order:
	1.	Header
	2.	Sticky actions
	3.	Assessment
	4.	Operational context
	5.	Needs action
	6.	Evidence
	7.	Work + procurement
	8.	Diagnostics
	9.	Activity timeline
	10.	Related tools

⸻

If Existing Code Already Exists

If there is already:
	•	equipment-intelligence.tsx
	•	a detail drawer
	•	related PdM components

Then do this:
	1.	Reuse useful business logic and data hooks
	2.	Stop using the drawer as the primary interaction model
	3.	Convert the detail experience into a full page or page-level route
	4.	Reduce duplication between “overview” and “prediction”
	5.	Eliminate or avoid redundant chat / AI sections
	6.	Add operational context and linked procurement visibility

⸻

Deliverables
	1.	Implement the page UI
	2.	Wire it to realistic mock or existing data sources
	3.	Keep code organized and readable
	4.	Add clear TODO comments where real backend integration is still needed
	5.	Briefly summarize:
	•	what was built
	•	what old patterns were removed or reduced
	•	what still needs backend support

Focus on making the page feel like the operational home for one equipment item.
