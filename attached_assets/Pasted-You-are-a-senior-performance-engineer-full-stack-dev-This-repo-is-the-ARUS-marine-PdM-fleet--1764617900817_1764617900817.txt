PHASE 2 — FRONTEND LAG & RENDER OPTIMIZATION



Goal: Stop unnecessary re-renders and cut UI jank.

Identify heavy components:

Fleet overview / bridge view / equipment health dashboards

Telemetry visualizations

Engine & deck logbooks (big tables)

Work orders + inventory lists

Any page using large tables or nested lists

For each heavy component:

Check for:

Props changing unnecessarily

Inline functions/objects causing re-renders

State stored too high in the tree

Large arrays mapped on every render without memoization

Apply safe optimizations:

Use React.memo for pure presentational components

Use useMemo / useCallback where it genuinely reduces work

Split large components into smaller subcomponents (e.g. per-row, per-card)

For 24-row or 100+ row tables, consider:

Virtualized lists (react-window or similar) OR

Pagination + “load more” patterns

React Query:

Audit all queries for:

Overly short staleTime / refetchInterval

Unnecessary refetchOnWindowFocus or refetchOnReconnect

Multiple components querying the same endpoint redundantly

Optimize:

Increase staleTime for data that doesn’t need to update every few seconds

Avoid polling where WebSockets/telemetry streams are already pushing updates

Centralize shared queries instead of duplicating them per component

Telemetry in UI:

Ensure telemetry updates only affect small, focused components, not entire pages.

Use:

Localized state per widget

Batched updates

Throttling/debouncing for UI updates (e.g. update chart every 1–5 seconds, not every packet)



Document all frontend changes in docs/perf-frontend-changes.md.

For each slow endpoint:
