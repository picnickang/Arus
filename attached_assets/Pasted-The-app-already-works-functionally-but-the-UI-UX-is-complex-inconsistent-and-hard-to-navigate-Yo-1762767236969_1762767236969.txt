2. High-level UX goals



Redesign the app to feel like a modern, focused PdM platform similar to the best tools in the industry.



Global goals

Clarity: Every screen answers: “What am I looking at?” and “What should I do next?”

Hierarchy: Important info (risk, health, alarms, next actions) is visible above less critical details.

Consistency: Reuse patterns (Cards, Tabs, Status badges, tables, dialogs, wizards).

Low cognitive load: Hide advanced configuration behind expandable panels or “Advanced” sections.

Safe operations: Destructive actions are clearly labeled, require confirmation, and explain impact.



Specific UX pains to fix

Oversized, dense pages (like the current Equipment Registry) with too many dialogs and forms in one component.

Sensor templates UX that is confusing and “expert-only.”

Mixed layouts where tables, pagination, and controls are scattered or visually ungrouped.

Inconsistent labels, tooltips, and status indicators.

5. General cleanup & code quality



While doing this UI/UX overhaul, also:

Eliminate JSX syntax errors

Ensure all ternary branches and && conditionals return exactly one JSX root.

Wrap siblings in fragments (<>...</>) when needed.

Fix any “Unexpected token, expected ‘,’” type issues from malformed JSX or trailing commas in the wrong place.

Break down huge components

Any React component file over ~500 lines should be split into smaller presentational subcomponents.

Move repeated patterns into reusable components (e.g. StatusBadge, EmptyState, SensorStatusRow).

Keep data types correct

Respect existing TypeScript types from @shared/schema.

Do not change back-end API shapes.

If you need new UI-only types, declare them locally or in a UI types file.

Keep existing test IDs

Do not remove data-testid attributes that exist (they’re used for testing).

If you add new elements, you may add new data-testids, but don’t rename existing ones without a strong reason.

Improve loading & error states

For every data-fetching screen (TanStack Query), ensure:

A visible loading state (spinner or “Loading …” text).

A clear error state with a retry button if something fails.
